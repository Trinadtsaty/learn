## 1.1. Установка Go

### Где скачать?
Официальный сайт: [https://go.dev/dl/](https://go.dev/dl/).  
Там представлены установщики для всех основных операционных систем (Windows, macOS, Linux). Всегда выбирайте последнюю стабильную версию (на момент написания это go1.22.x). Скачивайте файл, соответствующий вашей системе.
### Установка на разные ОС
#### **Windows**
1. Скачайте `.msi` файл (например, `go1.22.0.windows-amd64.msi`).
2. Запустите его и следуйте инструкциям мастера установки. По умолчанию Go устанавливается в `C:\Go`.
3. Установщик автоматически добавит `C:\Go\bin` в системную переменную `PATH`. После установки **обязательно перезапустите командную строку** (или PowerShell), чтобы изменения вступили в силу.
#### **macOS**
Есть два основных способа:
- **Скачать .pkg пакет** с официального сайта. Запустите его и следуйте инструкциям. Он установит Go в `/usr/local/go` и добавит `/usr/local/go/bin` в `PATH` (обычно через файл `/etc/paths.d/go`).
- **Через Homebrew** (если он у вас установлен):
```bash
brew install go
```
   Homebrew также сам пропишет путь.
#### **Linux**
1. Скачайте архив `.tar.gz` (например, `go1.22.0.linux-amd64.tar.gz`).
2. Откройте терминал и выполните следующие команды (понадобятся права sudo):
```bash
sudo tar -C /usr/local -xzf go1.22.0.linux-amd64.tar.gz
```
3. Теперь нужно добавить `/usr/local/go/bin` в переменную `PATH`. Для этого отредактируйте файл `~/.profile` или `~/.bashrc` (если используете bash). Добавьте в конец строку:
```bash
export PATH=$PATH:/usr/local/go/bin
```  
    Затем примените изменения:
```bash
source ~/.profile   # или source ~/.bashrc
```
### Проверка установки
После установки откройте терминал (командную строку) и выполните:
```bash
go version
```
Вы должны увидеть что‑то вроде:
```text
go version go1.22.0 linux/amd64
```

Если команда не найдена — значит, Go либо не установлен, либо не добавлен в `PATH`. Переходим к разбору `PATH`.
## Что такое PATH и как его проверить/настроить?
**PATH** — это переменная окружения, в которой через разделитель (двоеточие на Linux/macOS, точка с запятой на Windows) перечислены пути к каталогам, где операционная система ищет исполняемые файлы. Когда вы вводите команду `go`, система просматривает все эти каталоги в поисках программы с именем `go` (или `go.exe` на Windows). Если каталог с Go не в списке, команда не будет найдена.
### Как проверить текущее значение PATH
- **Windows** (командная строка):
```cmd
echo %PATH%
```
   Или в PowerShell:
   ```powershell
   $env:Path
   ```
- **Linux / macOS**:
```bash
echo $PATH
```
Ищите среди вывода путь, содержащий `go` (обычно `C:\Go\bin` на Windows, `/usr/local/go/bin` на Linux/macOS). Если его нет — нужно добавить вручную.
### Как добавить Go в PATH вручную (если автоматически не добавилось)
#### **Windows**
1. Откройте «Система» → «Дополнительные параметры системы» → «Переменные среды».
2. В разделе «Системные переменные» найдите переменную `Path`, выберите её и нажмите «Изменить».
3. Нажмите «Создать» и добавьте `C:\Go\bin` (или путь, куда вы установили Go).
4. Нажмите «Ок» во всех окнах и перезапустите командную строку.
#### **macOS / Linux**
Если автоматическая установка не сработала, можно добавить путь в конфигурационный файл вашей оболочки. Например, для `bash`:
```bash
echo 'export PATH=$PATH:/usr/local/go/bin' >> ~/.bashrc
source ~/.bashrc
```
Для `zsh` (по умолчанию на новых macOS):
```bash
echo 'export PATH=$PATH:/usr/local/go/bin' >> ~/.zshrc
source ~/.zshrc
```
После этого снова проверьте `go version`.
## Первая программа на Go
Теперь, когда Go установлен и доступен из командной строки, напишем первую программу.
### Создание рабочей директории и файла
Создайте отдельную папку для проекта, например, `hello`. Внутри неё создайте файл `main.go`. Расширение `.go` говорит о том, что это исходный код на Go.
### Код программы
Откройте `main.go` в любом текстовом редакторе (можно даже в Блокноте) и напишите следующий код:
```go
package main
import "fmt"
func main() {
    fmt.Println("Hello, Go!")
}
```
**Разберём каждую строку:**
- `package main` — каждый файл `.go` начинается с объявления пакета. `package main` — это специальный пакет, который говорит компилятору, что мы создаём исполняемую программу (а не библиотеку). Именно в пакете `main` должна находиться функция `main`, с которой начинается выполнение.
- `import "fmt"` — мы подключаем стандартный пакет `fmt` (сокращение от «format»), который содержит функции для форматированного ввода/вывода, например, `Println` для вывода строки.
- `func main() { ... }` — это точка входа в программу. Когда вы запускаете скомпилированный файл, операционная система вызывает функцию `main`. Внутри неё мы вызываем `fmt.Println("Hello, Go!")`, которая выводит текст и переводит строку.
### Запуск программы без компиляции (go run)
Go позволяет быстро запустить программу, не создавая отдельного исполняемого файла. Для этого используется команда:
```bash
go run main.go
```
Вы должны увидеть вывод:
```text
Hello, Go!
```
Что происходит под капотом: `go run` компилирует ваш код во временный исполняемый файл, запускает его, а затем удаляет. Это удобно для тестирования небольших программ.
### Компиляция в исполняемый файл (go build)
Чтобы получить самостоятельный бинарник, который можно запускать на любой машине с такой же архитектурой (без установленного Go), используйте:
```bash
go build main.go
```
После выполнения этой команды в текущей папке появится исполняемый файл:
- На Windows: `main.exe`
- На Linux/macOS: `main`
Теперь его можно запустить:
- **Windows**: `main.exe`
- **Linux/macOS**: `./main`
Программа выведет то же самое:
```text
Hello, Go!
```
**Что такое компиляция?**  
Компиляция — это процесс перевода исходного кода, написанного человеком на языке программирования (Go), в машинный код (набор инструкций, понятный процессору). В результате получается исполняемый файл, который можно запускать напрямую, без интерпретатора или дополнительных сред.

## 1.2. Понимание GOPATH и модулей

### 1.2.1. GOPATH: исторический контекст
#### Что такое GOPATH?
В первых версиях Go (до 1.11) не было встроенной системы управления зависимостями с версионированием. Весь код, включая сторонние библиотеки, должен был находиться внутри единой рабочей области, путь к которой указывался в переменной окружения `GOPATH`. По умолчанию `GOPATH` указывал на директорию `go` в домашней папке пользователя (например, `~/go` на Linux/macOS или `%USERPROFILE%\go` на Windows).
Структура `GOPATH` была строго определена и включала три подпапки:
- **src** — исходный код. Здесь лежали все проекты и сторонние пакеты. Путь внутри `src` соответствовал импорту. Например, если вы писали проект `github.com/user/project`, он должен был лежать в `$GOPATH/src/github.com/user/project`. Сторонние пакеты при установке (`go get`) также скачивались сюда.
- **pkg** — собранные объектные файлы (архивы `.a`). Использовались для ускорения компиляции.
- **bin** — скомпилированные исполняемые файлы. Сюда `go install` помещал бинарники, и эту папку часто добавляли в `PATH`.
Пример:
```text
$GOPATH/
  bin/
    hello               # скомпилированная программа
  pkg/
    linux_amd64/        # платформозависимые объекты
      github.com/user/
        mylib.a
  src/
    github.com/user/
      hello/
        main.go         # ваш проект
      mylib/
        mylib.go        # библиотека
```
#### Почему отказались от GOPATH?
GOPATH имел серьёзные недостатки:
1. **Отсутствие версионирования зависимостей**. Если два проекта требовали разные версии одной библиотеки, они не могли сосуществовать в одном GOPATH. Приходилось использовать утилиты вроде `godep`, `govendor` для копирования кода в папку `vendor` внутри проекта.
2. **Воспроизводимость сборок**. Без фиксации версий невозможно было гарантировать, что сборка проекта через месяц будет использовать те же версии библиотек.
3. **Жёсткая привязка к структуре папок**. Весь код должен был лежать в GOPATH, что мешало размещать проекты в произвольных местах файловой системы.
4. **Сложность работы с форками и локальными версиями**. Чтобы использовать модифицированную версию библиотеки, приходилось вручную подменять файлы.
Всё это привело к созданию Go Modules — официальной системы управления зависимостями, которая появилась в Go 1.11 (экспериментально) и стала стандартом с Go 1.13.

### 1.2.2. Go Modules
#### Что такое модуль?
**Модуль** — это набор Go-пакетов, объединённых общим корнем и файлом `go.mod`, который определяет:
- путь модуля (обычно репозиторий, например, `github.com/user/project`);
- версию Go, с которой совместим модуль;
- зависимости (другие модули) с их версиями.
Модуль может находиться в любой папке, не обязательно внутри GOPATH. Это ключевое отличие — теперь вы можете работать над проектом где угодно.
#### Инициализация модуля: `go mod init`
Для создания нового модуля используется команда:
```bash
go mod init <module name>
```
`<module name>` — это обычно путь к репозиторию, где будет храниться код (например, `github.com/username/project`). Это имя будет использоваться для импорта пакетов внутри модуля и другими модулями.
Пример:
```bash
mkdir myproject
cd myproject
go mod init github.com/alice/myproject
```
После выполнения создаётся файл `go.mod` с содержимым:
```text
module github.com/alice/myproject

go 1.22
```
#### Файл `go.mod` и его директивы
`go.mod` — текстовый файл, который описывает модуль. Основные директивы:
- **module** — путь модуля.
- **go** — минимальная версия Go, требуемая для сборки.
- **require** — список зависимостей с версиями. Добавляется автоматически при использовании `go get` или вручную.
- **replace** — замена модуля на другой локальный путь или другую версию. Полезно для локальной разработки.
- **exclude** — исключение конкретной версии модуля.
Пример `go.mod` после добавления зависимости:
```text
module github.com/alice/myproject

go 1.22

require github.com/google/uuid v1.3.0
```
#### Файл `go.sum`

`go.sum` содержит контрольные суммы (хэши) каждой версии зависимостей. Он обеспечивает **целостность и безопасность**: при повторной сборке Go проверяет, что скачанные файлы соответствуют ожидаемым хэшам, защищая от подмены или повреждения.

`go.sum` включает записи для каждой версии модуля и для каждого его файла. Пример:
```text
github.com/google/uuid v1.3.0 h1:...
github.com/google/uuid v1.3.0/go.mod h1:...
```
#### Версионирование зависимостей (Semantic Versioning, semver)

Go использует семантическое версионирование (semver) — формат `vMAJOR.MINOR.PATCH` (например, `v1.3.0`).
- **MAJOR** — несовместимые изменения API.
- **MINOR** — новая функциональность, обратно совместимая.
- **PATCH** — исправления багов, обратно совместимые.
При добавлении зависимости Go выбирает **наибольшую доступную совместимую версию** согласно правилам семвера. Для этого используется алгоритм **минимальной версии (Minimal Version Selection, MVS)**:
- Go берёт максимальную версию, требуемую любым из модулей в графе зависимостей, но не выше той, что явно указана.
- Это гарантирует детерминированность сборки.
#### Команды для работы с модулями
##### `go get`
`go get` используется для добавления, обновления или понижения версий зависимостей. Основные варианты:
- `go get github.com/google/uuid` — добавить последнюю версию.
- `go get github.com/google/uuid@v1.2.0` — получить конкретную версию.
- `go get -u` — обновить все зависимости до минорных или патч-версий.
- `go get -u ./...` — обновить все зависимости проекта.
- `go get -d` — только скачать, но не устанавливать (устарело, сейчас `go get` по умолчанию только добавляет в `go.mod`).
После выполнения `go get` обновляются `go.mod` и `go.sum`.
##### `go mod tidy`
`go mod tidy` приводит файлы модуля в порядок:
- добавляет недостающие модули, необходимые для сборки текущего пакета и его тестов;
- удаляет неиспользуемые зависимости.

Это рекомендуемая команда после написания кода, чтобы синхронизировать `go.mod` с реальными импортами.
##### `go mod vendor`
`go mod vendor` создаёт папку `vendor` в корне проекта и копирует в неё исходный код всех зависимостей. Раньше этот подход использовался для фиксации зависимостей, но с появлением модулей он не обязателен. Однако он может быть полезен:
- для обеспечения воспроизводимости сборки в средах без доступа к интернету;
- для некоторых CI-систем, где нужно всё хранить в репозитории.
При использовании `vendor` Go будет компилировать код из этой папки, а не из кеша модулей. Чтобы включить эту логику, нужно передать флаг `-mod=vendor` при сборке или тестировании (или по умолчанию, если папка `vendor` существует и вы используете Go 1.14+).
##### `go mod replace`
Директива `replace` в `go.mod` позволяет подменить путь к модулю. Например, если вы разрабатываете две библиотеки локально и хотите, чтобы одна использовала ещё не опубликованную версию другой:
```text
replace github.com/old/path => ../local/path
```
После этого все импорты `github.com/old/path` будут перенаправляться на локальную папку. Это удобно для разработки без необходимости публиковать изменения.
#### Типичный рабочий процесс с модулями
1. Создаёте папку проекта и инициализируете модуль:
```bash
mkdir myapp
cd myapp
go mod init github.com/me/myapp
```
2. Пишете код, импортируете внешние пакеты.
3. Добавляете зависимости:
```bash
go get github.com/google/uuid
```
или просто запускаете `go build`, и Go автоматически подтянет недостающие модули (но лучше явно добавить через `go get`).
4. После завершения работы запускаете `go mod tidy`, чтобы очистить `go.mod`.
5. Если нужно зафиксировать зависимости в папке `vendor` (например, для CI без доступа в интернет), выполняете `go mod vendor`.
6. Коммитите `go.mod`, `go.sum` и (опционально) `vendor/` в систему контроля версий.

```bash
cd uuid-example
```
2. **Инициализируйте модуль:**
```bash
go mod init github.com/yourusername/uuid-example
```
Вместо `yourusername` подставьте свой логин на GitHub (или любой другой идентификатор). Это не обязательно должен быть реальный URL, но это хорошая практика.

После выполнения появится файл `go.mod`:
```text
module github.com/yourusername/uuid-example

go 1.22
```
3. **Напишите программу `main.go`:**  
Создайте файл `main.go` со следующим содержимым:
```go
package main

import (
    "fmt"
    "github.com/google/uuid"
)

func main() {
    id := uuid.New()
    fmt.Printf("Generated UUID: %s\n", id)
}
```
4. **Добавьте зависимость:**  
Выполните команду:
```bash
go get github.com/google/uuid
```
Вывод покажет, какая версия выбрана (например, `v1.3.0`). Теперь откройте `go.mod` — вы увидите новую секцию `require`:
```text
module github.com/yourusername/uuid-example

go 1.22

require github.com/google/uuid v1.3.0
```
Также появился файл `go.sum` с контрольными суммами.
5. **Запустите программу:**
```bush
go run main.go
```
Должен вывестись сгенерированный UUID, например:
```text
Generated UUID: 123e4567-e89b-12d3-a456-426614174000
```
6. **Выполните `go mod tidy`:**
```bash
go mod tidy
```
Команда проверит, все ли импорты покрыты, и удалит лишние зависимости. В нашем случае ничего не изменится, но это полезно для порядка.
7. **Выполните `go mod vendor`:**
```bash
go mod vendor
```
После этого появится папка `vendor/`, содержащая копию исходного кода `github.com/google/uuid` (и любых других зависимостей). Структура будет похожа на:
```text
vendor/
  github.com/
    google/
      uuid/
        ... (файлы библиотеки)
  modules.txt
  
```
Теперь можно собрать программу с использованием vendor:
```bash
go build -mod=vendor -o uuid-app
```
или просто `go build`, так как Go 1.14+ автоматически использует vendor, если папка существует.
8. **Проверьте итоговые файлы:**
    - `go.mod` содержит модуль и require.
    - `go.sum` содержит хэши.
    - `vendor/` содержит код зависимостей.
Теперь вы можете закоммитить эти файлы (кроме, возможно, `vendor/`, если не хотите хранить его в репозитории). Это основа работы с модулями в Go.

**Что произошло под капотом:**
- `go get` скачал модуль `github.com/google/uuid` версии `v1.3.0` в кеш модулей (обычно `$GOPATH/pkg/mod`).
- В `go.mod` добавилась запись require.
- В `go.sum` записались хэши для проверки целостности.
- `go mod vendor` скопировал файлы из кеша в папку `vendor`.

## 1.3. Первая программа на Go
Теперь, когда мы разобрались с GOPATH и модулями, создадим нашу первую полноценную программу. Это фундамент, на котором будет строиться всё дальнейшее понимание языка.
### 1.3.1. Создание файла main.go
#### Где создавать файл?
Мы уже создали модуль `uuid-example` в прошлом уроке. Для новой программы создадим отдельный модуль, чтобы не смешивать примеры.
```bash
mkdir myfirstprog
cd myfirstprog
go mod init example/myfirstprog
```
Теперь в этой папке создайте файл `main.go`. **Расширение `.go`** обязательно — оно сообщает компилятору и редакторам кода, что это исходный файл на языке Go.
### 1.3.2. Структура программы
Откройте `main.go` и напишите следующий код:
```go
package main

import "fmt"

func main() {
    fmt.Println("Привет, мир!")
}
```
Разберём каждую строку подробно:
#### Пакет `main` — точка входа
```go
package main
```
**Что такое пакет?**  
Пакет (package) — это способ организации и группировки связанных файлов Go. Каждый файл `.go` начинается с объявления пакета.

**Почему именно `main`?**  
В Go есть специальный пакет с именем `main`. Он указывает компилятору, что мы создаём **исполняемую программу**, а не библиотеку. Исполняемая программа — это файл, который можно запустить самостоятельно (например, `.exe` на Windows или бинарный файл на Linux). Библиотека же предназначена для использования в других программах.

Если вы назовёте пакет иначе (например, `package mylib`), то после компиляции вы получите библиотечный файл (`.a`), который нельзя запустить напрямую.
#### Функция `main()` — сердце программы
```go
func main() {
    // код
}
```
`func main()` — это **точка входа** в программу. Когда операционная система запускает скомпилированный файл, она ищет именно эту функцию и начинает выполнение с неё. Без функции `main` в пакете `main` компилятор выдаст ошибку: `runtime.main_main·f: function main is undeclared in the main package`.

Внутри фигурных скобок `{}` пишется код, который должен выполниться при запуске. В нашем случае это одна строка вывода.
#### Импорт пакета `fmt`
```go
import "fmt"
```
`import` подключает другие пакеты, чтобы использовать их функции. Здесь мы импортируем стандартный пакет `fmt` (от слова format), который содержит функции для форматированного ввода/вывода.
#### Функция `fmt.Println()`
```go
fmt.Println("Привет, мир!")
```
- `fmt.` — обращение к импортированному пакету.
- `Println` — функция внутри пакета `fmt`, которая выводит переданные аргументы в консоль и добавляет перевод строки (ln = line).
- `"Привет, мир!"` — строковый литерал (последовательность символов в кавычках).
### 1.3.3. Запуск программы: `go run`

Чтобы сразу увидеть результат, используйте команду:
```bash
go run main.go
```
Вывод:
```text
Привет, мир!
```
**Что происходит под капотом?**  
Команда `go run` делает следующее:

1. Компилирует ваш код во временный исполняемый файл (обычно в системной временной папке, например, `/tmp/go-build...` на Linux).
2. Запускает этот временный файл.
3. После завершения программы удаляет временный файл.

Это идеальный инструмент для разработки: вы быстро проверяете код, не засоряя папку проекта лишними файлами.

### 1.3.4. Компиляция: `go build`

Когда программа готова и вы хотите получить самостоятельный исполняемый файл, используйте:
```bash
go build
```
или явно указав файл:
```bash
go build main.go
```
**Что происходит?**  
Компилятор Go (это программа `go`, которая вызывает компилятор) транслирует ваш исходный код в машинный код — набор инструкций, понятных процессору вашего компьютера. Результат — исполняемый файл.

**Где появляется файл?**
- Если вы выполнили `go build` без аргументов, файл появится в текущей папке и будет называться так же, как папка проекта (в нашем случае `myfirstprog` на Linux/macOS или `myfirstprog.exe` на Windows).
- Если вы указали `go build main.go`, то файл будет называться `main` (или `main.exe`).

Теперь вы можете запустить его:
- **Linux/macOS**:
```bash
./myfirstprog
```
- **Windows**:
```cmd
myfirstprog.exe
```
Программа выведет то же самое: `Привет, мир!`.

#### Почему исполняемый файл такой большой?

Если вы посмотрите на размер файла (например, командой `ls -lh` на Linux), то увидите, что даже простая программа весит около 1–2 МБ. Это нормально! Go использует **статическую компиляцию**: все необходимые библиотеки (включая рантайм Go — сборщик мусора, планировщик горутин и т.д.) встраиваются прямо в исполняемый файл. Благодаря этому программу можно запустить на любом компьютере с той же архитектурой, даже если там не установлен Go. Никаких внешних зависимостей (DLL, so-файлов) не требуется.
### 1.3.5. Разница между `go run` и `go build`

|Команда|Назначение|Результат|Когда использовать|
|---|---|---|---|
|`go run`|Быстрый запуск для тестирования|Временный файл, удаляется после запуска|В процессе разработки, для проверки|
|`go build`|Создание готового исполняемого файла|Постоянный файл в текущей папке|Когда нужно распространять программу|

**Полный цикл разработки:**
1. Пишете код в `main.go`.
2. Тестируете: `go run main.go`.
3. Если есть ошибки — исправляете, снова `go run`.
4. Когда всё работает — компилируете: `go build`.
5. Запускаете готовый бинарник: `./myfirstprog`.

## 1.4. Редактор кода / IDE и Практика #1

Теперь, когда мы научились создавать и запускать простые программы, пришло время настроить среду разработки. Правильный инструмент значительно ускоряет написание кода, помогает избегать ошибок и делает процесс обучения комфортным. В этом разделе мы рассмотрим популярные редакторы и IDE для Go, подробно настроим VS Code (рекомендуемый для начинающих) и выполним первое комплексное практическое задание, которое объединит всё, что мы прошли в разделе 1.

### 1.4.1. Зачем нужен редактор кода / IDE?

Редактор кода (или IDE — Integrated Development Environment) — это программа для написания и редактирования исходного кода. Хороший редактор предоставляет:
- **Подсветку синтаксиса** — разные элементы кода окрашиваются в разные цвета, что облегчает чтение.
- **Автодополнение (IntelliSense)** — предлагает варианты завершения имён функций, переменных, пакетов.
- **Форматирование кода** — автоматически приводит код к единому стилю.
- **Статический анализ (линтер)** — подсвечивает потенциальные ошибки и несоответствия лучшим практикам.
- **Отладку** — позволяет ставить точки останова, смотреть значения переменных.
- **Интеграцию с системой контроля версий** (Git) и терминалом.

### 1.4.2. Выбор редактора/IDE для Go

#### **VS Code** (Visual Studio Code)
- **Бесплатный**, с открытым исходным кодом, кроссплатформенный.
- Лёгкий, но расширяемый через плагины.
- Официальное расширение Go от команды Go обеспечивает полную поддержку языка.
- Идеален для новичков: просто установить, много обучающих материалов.
#### **GoLand**
- Платная IDE от JetBrains (есть бесплатная лицензия для студентов и преподавателей).
- Мощный инструмент «всё в одном»: глубокая интеграция с Go, рефакторинг, работа с базами данных, поддержка популярных фреймворков.
- Требовательна к ресурсам, но предоставляет максимальный комфорт.
#### **Альтернативы**
- **Vim/Neovim** с плагинами (vim-go, coc.nvim) — для тех, кто уже знаком с Vim.
- **Sublime Text** с плагинами Go.
- **Eclipse** с плагином GoClipse.

**Для новичка я рекомендую VS Code** — он прост в освоении, бесплатен и даёт всё необходимое. GoLand можно рассмотреть позже, когда вы освоитесь и, возможно, получите студенческую лицензию.

### 1.4.3. VS Code: установка и настройка

#### Шаг 1: Скачайте и установите VS Code

Перейдите на [официальный сайт](https://code.visualstudio.com/) и скачайте установщик для вашей ОС. Установка стандартная.

#### Шаг 2: Установите расширение Go
- Запустите VS Code.
- Откройте панель расширений (Ctrl+Shift+X или значок квадратиков слева).
- В поиске введите `Go`.
- Найдите расширение **Go** от команды Go (автор — Go Team at Google) и нажмите «Install».

#### Что входит в расширение Go?

Расширение включает в себя несколько ключевых инструментов, которые устанавливаются автоматически (или по запросу):
- **`gopls`** — языковой сервер (Language Server Protocol). Он обеспечивает автодополнение, переход к определению, поиск ссылок, рефакторинг и многое другое. Работает «под капотом», общаясь с редактором.
- **`dlv`** (Delve) — отладчик для Go.
- **`gofmt`**, **`goimports`** — инструменты форматирования кода.
- **`staticcheck`**, **`golint`** — линтеры (статический анализ).
- **`go-outline`, `guru`** и другие вспомогательные утилиты.

#### Шаг 3: Проверьте, что расширение работает
- Откройте VS Code в любой папке (например, создайте новую папку `test` и откройте её через File > Open Folder).
- Создайте файл `main.go`.
- Начните печатать `package main`. Вы должны увидеть подсказки автодополнения.
- Расширение может предложить установить недостающие инструменты. Если появится уведомление в правом нижнем углу, нажмите «Install All». Это скачает и установит все необходимые утилиты через `go get`.

### 1.4.4. Настройка автодополнения, форматирования, линтинга

VS Code с расширением Go имеет разумные настройки по умолчанию, но их можно уточнить для более комфортной работы.

#### Автодополнение

Управляется через `gopls`. Обычно не требует настройки. Если хотите убедиться, что используется `gopls`, откройте настройки (Ctrl+,) и найдите `go.useLanguageServer` — должно быть `true`.

#### Форматирование при сохранении

Очень полезная функция: код автоматически форматируется каждый раз, когда вы сохраняете файл.

- По умолчанию используется `gofmt`. Можно выбрать `goimports` (он дополнительно сортирует и добавляет/удаляет импорты).
- Чтобы включить форматирование при сохранении:
    1. Откройте настройки (Ctrl+,).
    2. В поиске введите `format on save`.
    3. Установите галочку **Editor: Format On Save**.
    4. Чтобы убедиться, что форматирование Go использует `goimports`, найдите `go.formatTool` и выберите `goimports` (по умолчанию `gofmt`).

#### Линтинг

Линтер проверяет код на потенциальные проблемы и стилистические ошибки.

- Расширение Go может использовать `staticcheck` (рекомендуется) или `golint`.
- По умолчанию линтер запускается при сохранении. Вы можете настроить его в `go.lintTool` (выбрать `staticcheck`) и `go.lintFlags`.
- Если линтер не установлен, расширение предложит установить его.

#### Возможные проблемы
Иногда расширение не может найти установленные инструменты. В таком случае можно установить их вручную через терминал:
```bash
go install golang.org/x/tools/gopls@latest
go install github.com/go-delve/delve/cmd/dlv@latest
go install golang.org/x/tools/cmd/goimports@latest
go install honnef.co/go/tools/cmd/staticcheck@latest
```
После установки перезапустите VS Code.

8. **Измените программу**, чтобы она выводила результат функции `quote.Go()` из этой библиотеки.
9. **Запустите программу** через `go run` и убедитесь, что она работает.
10. **Скомпилируйте программу** в исполняемый файл через `go build` и запустите полученный бинарник.
11. **Убедитесь, что форматирование при сохранении работает**: внесите беспорядочные отступы, сохраните файл — они должны исправиться.
12. **Изучите файлы `go.mod` и `go.sum`** — посмотрите, какие версии зависимостей добавились.

#### Решение (пошагово с комментариями)

##### Шаг 1: Проверка установки Go

Откройте терминал (в VS Code можно через Ctrl+` или меню Terminal > New Terminal) и выполните:
```bash
go version
```
Если видите версию (например, `go version go1.22.0 linux/amd64`) — всё готово.
##### Шаг 2: Создание папки и открытие в VS Code
В терминале:
```bash
mkdir hello-quote
cd hello-quote
code .   # Открывает текущую папку в VS Code (если команда code настроена)
```
Если команда `code` не работает, откройте VS Code вручную и выберите File > Open Folder, укажите созданную папку.
##### Шаг 3: Инициализация модуля

В терминале VS Code выполните:
```bash
go mod init hello-quote
```
Создастся файл `go.mod` примерно такого содержания:
```text
module hello-quote

go 1.22
```
(версия `go` может отличаться в зависимости от вашей установки)
##### Шаг 4: Создание `main.go`
В VS Code создайте новый файл `main.go` (через иконку нового файла в проводнике). Напишите первоначальный код:
```go
package main

import "fmt"

func main() {
    fmt.Println("Hello, world!")
}
```
Сохраните файл (Ctrl+S). Обратите внимание, что при сохранении код отформатируется (например, добавит табуляцию).
##### Шаг 5: Добавление внешней библиотеки

В терминале выполните:
```bash
go get rsc.io/quote
```
Вывод будет примерно таким:
```text
go: downloading rsc.io/quote v1.5.2
go: downloading rsc.io/sampler v1.3.0
go: downloading golang.org/x/text v0.0.0-20170915032832-14c0d48ead0c
```
Что произошло:
- `go get` скачал библиотеку `rsc.io/quote` и её транзитивные зависимости (`rsc.io/sampler`, `golang.org/x/text`).
- Обновился `go.mod` — добавилась строка `require rsc.io/quote v1.5.2`.
- Создался (или обновился) файл `go.sum`, содержащий контрольные суммы для проверки целостности.
##### Шаг 6: Изменение программы
Отредактируйте `main.go`:
```go
package main

import "fmt"
import "rsc.io/quote"

func main() {
    fmt.Println(quote.Go())
}
```
Обратите внимание: при импорте нового пакета `rsc.io/quote` расширение Go автоматически добавит его, если вы выберете соответствующее действие в подсказке, или можно написать вручную. После сохранения `goimports` упорядочит импорты:
```go
package main

import (
    "fmt"
    "rsc.io/quote"
)

func main() {
    fmt.Println(quote.Go())
}
```
##### Шаг 7: Запуск через `go run`
В терминале:
```bash
go run main.go
```
Вы должны увидеть знаменитую цитату:
```text
Don't communicate by sharing memory, share memory by communicating.
```
(это одна из ключевых идиом Go, возвращаемая функцией `quote.Go()`)
##### Шаг 8: Компиляция в исполняемый файл
Выполните:
```bash
go build
```
Появится исполняемый файл `hello-quote` (или `hello-quote.exe` на Windows). Запустите его:
- Linux/macOS: `./hello-quote`
- Windows: `hello-quote.exe`

Вывод будет таким же, как при `go run`.
##### Шаг 9: Проверка форматирования при сохранении
Намеренно сделайте в `main.go` отступы хаотичными, например:
```go
package main

import ( "fmt";"rsc.io/quote"
)

func main() {
fmt.Println(quote.Go())}
```
Сохраните файл (Ctrl+S). Он должен автоматически отформатироваться в принятом стиле Go.
##### Шаг 10: Изучение `go.mod` и `go.sum`

Откройте файл `go.mod`:
```text
module hello-quote

go 1.22

require rsc.io/quote v1.5.2
```
Обратите внимание, что версия `v1.5.2` указана с префиксом `v` — это соответствует семантическому версионированию.

Файл `go.sum` содержит записи для каждой версии каждого модуля:
```text
rsc.io/quote v1.5.2 h1:...
rsc.io/quote v1.5.2/go.mod h1:...
rsc.io/sampler v1.3.0 h1:...
...
```
Эти хэши гарантируют, что при повторной загрузке зависимости будут теми же самыми.
