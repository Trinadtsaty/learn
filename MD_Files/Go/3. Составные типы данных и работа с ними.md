## 3.1. Массивы

Массивы — это фундаментальный составной тип данных в Go. Они представляют собой последовательность элементов одного типа фиксированной длины. Понимание массивов важно, так как они лежат в основе более гибких слайсов, которые мы будем изучать далее.
### 3.1.1. Объявление массивов

#### Синтаксис и нулевые значения

Массив объявляется с указанием размера в квадратных скобках, затем типа элементов:
```go
var arr [5]int  // массив из 5 целых чисел, все элементы инициализированы нулями
```
При объявлении без явной инициализации все элементы получают **нулевое значение** для своего типа:

- Для `int`: `0`
- Для `string`: `""`
- Для `bool`: `false`

#### Инициализация с литералом
```go
arr1 := [3]int{1, 2, 3}           // массив из 3 элементов
arr2 := [5]int{1, 2, 3}           // первые три элемента: 1,2,3, остальные: 0,0
arr3 := [...]int{1, 2, 3}         // ... — компилятор сам подсчитает длину (3)
arr4 := [5]int{0: 10, 2: 30}      // индексы 0 и 2 инициализированы, остальные — 0
```
**Важно:** `[...]` — это не "бесконечный массив", а указание компилятору вычислить размер по количеству элементов в литерале. Размер становится фиксированным на этапе компиляции.

### 3.1.2. Доступ к элементам

Доступ к элементам осуществляется по индексу (от 0 до `len(arr)-1`):
```go
arr := [3]int{10, 20, 30}
arr[0] = 100          // изменение элемента
value := arr[1]       // чтение: 20

fmt.Println(len(arr)) // 3 — длина массива (константа времени компиляции)
```
**Выход за границы:** при попытке обратиться к индексу, выходящему за пределы, программа вызовет **панику (panic)** во время выполнения:
```go
arr := [3]int{1, 2, 3}
arr[5] = 10  // panic: index out of range [5] with length 3
```
### 3.1.3. Передача массивов в функции

#### Передача по значению (копирование)

В Go массивы передаются в функции **по значению**. Это означает, что при вызове функции создаётся полная копия массива, включая все его элементы. Изменения внутри функции не влияют на исходный массив.
```go
func modifyArray(arr [3]int) {
    arr[0] = 100
    fmt.Println("Внутри функции:", arr) // [100, 20, 30]
}

func main() {
    original := [3]int{10, 20, 30}
    modifyArray(original)
    fmt.Println("После вызова:", original) // [10, 20, 30] — не изменился!
}
```
#### Передача указателя на массив

Чтобы изменения были видны снаружи, нужно передать **указатель** на массив:
```go
func modifyArrayPtr(arr *[3]int) {
    arr[0] = 100  // автоматическое разыменование (*arr)[0] = 100
    fmt.Println("Внутри функции:", *arr)
}

func main() {
    original := [3]int{10, 20, 30}
    modifyArrayPtr(&original)
    fmt.Println("После вызова:", original) // [100, 20, 30] — изменился!
}
```
#### Почему на практике массивы используются редко?

Копирование всего массива при передаче может быть дорогим для больших массивов. Кроме того, фиксированная длина часто неудобна. Именно поэтому в Go гораздо чаще используют **слайсы**, которые мы изучим в следующем разделе. Слайсы передаются по ссылке (как заголовок, указывающий на данные), что эффективнее и гибче.
### 3.1.4. Сравнение массивов

Массивы в Go можно сравнивать с помощью операторов `==` и `!=`, **если тип элементов сравним**.
```go
a := [3]int{1, 2, 3}
b := [3]int{1, 2, 3}
c := [3]int{1, 2, 4}

fmt.Println(a == b) // true
fmt.Println(a == c) // false
```
**Важно:** размер массива является частью его типа. Нельзя сравнивать массивы разной длины:
```go
// Ошибка компиляции: mismatched types [3]int and [4]int
// fmt.Println([3]int{1,2,3} == [4]int{1,2,3,4})
```
Если тип элемента несравним (например, `[]int` — слайс, который несравним), то и сам массив будет несравним.
### 3.1.5. Многомерные массивы

Go поддерживает многомерные массивы. Они представляют собой массивы массивов и также размещаются в памяти непрерывным блоком.
```go
var matrix [3][3]int  // 3x3 матрица, все элементы = 0

// Инициализация
matrix := [2][3]int{
    {1, 2, 3},
    {4, 5, 6},
}

// Доступ
matrix[0][1] = 10  // изменяет элемент во второй строке, втором столбце

// Длина
fmt.Println(len(matrix))    // 2 (количество строк)
fmt.Println(len(matrix[0])) // 3 (количество столбцов)
```
### 3.1.6. Как массивы работают "под капотом"

#### Размещение в памяти

Массив представляет собой **непрерывный блок памяти**. Например, `[5]int32` занимает 20 байт (5 × 4 байта) в памяти, и элементы расположены последовательно. Это обеспечивает предсказуемое размещение и хорошую локальность данных.

#### Размер — часть типа

В Go `[3]int` и `[5]int` — это **разные типы**, несовместимые друг с другом. Нельзя присвоить один другому или передать функцию, ожидающую `[5]int`, с аргументом `[3]int`. Это жёсткое ограничение, которое делает массивы менее гибкими, но даёт компилятору больше возможностей для оптимизации.

#### Компиляторные оптимизации

Для небольших массивов компилятор может:

- Размещать массив в регистрах, а не в стеке
- Полностью разворачивать циклы по известной длине
- Встраивать операции с массивами в код

Это делает работу с небольшими массивами очень эффективной.

#### Передача по значению и копирование

При передаче массива в функцию или присваивании массива другой переменной создаётся **побайтовая копия** всех элементов. Для больших массивов это может быть накладным.

## 3.2. Слайсы (самое важное!)

Слайсы — это самый важный составной тип в Go. Они предоставляют гибкий и эффективный способ работы с последовательностями данных. Понимание того, как слайсы работают "под капотом", критически важно для написания правильного и производительного кода.
### 3.2.1. Устройство слайса

Слайс — это **дескриптор** (заголовок), который ссылается на непрерывный участок памяти (базовый массив). Он состоит из трёх полей:
```go
type slice struct {
    ptr *array  // указатель на первый элемент базового массива
    len int     // длина слайса (количество доступных элементов)
    cap int     // ёмкость (максимальное количество элементов от ptr до конца массива)
}
```
**Визуализация:**
```text
Базовый массив: [10, 20, 30, 40, 50]
                     ↑
Слайс s := array[1:4] → ptr указывает на 20
                       len = 3 (20, 30, 40)
                       cap = 4 (20, 30, 40, 50)
```
### 3.2.2. Создание слайсов

#### Литерал слайса
```go
s := []int{1, 2, 3}  // len=3, cap=3
```
#### make — с указанием длины и ёмкости
```go
s1 := make([]int, 5)      // len=5, cap=5, элементы инициализированы нулями
s2 := make([]int, 3, 5)   // len=3, cap=5, доступно 3 элемента, резерв на 2
```
#### Срез массива или другого слайса
```go
arr := [5]int{10, 20, 30, 40, 50}
s := arr[1:4]  // [20, 30, 40], len=3, cap=4 (от индекса 1 до конца массива)
```
#### Нулевой (nil) слайс vs пустой слайс
```go
var s1 []int           // nil слайс (len=0, cap=0, ptr=nil)
s2 := []int{}          // пустой слайс (len=0, cap=0, ptr != nil, указывает на массив нулевой длины)
s3 := make([]int, 0)   // тоже пустой слайс
```
**Разница:** при сериализации в JSON `nil` слайс даёт `null`, а пустой — `[]`. В остальном оба можно использовать в `append`, `len`, `cap`. Обычно предпочитают `nil` слайс, если не нужен пустой литерал.

### 3.2.3. Оператор среза

#### Простая форма `[low:high]`
```go
arr := [5]int{10, 20, 30, 40, 50}
s := arr[1:4]   // элементы 1, 2, 3 (индексы 1,2,3)
fmt.Println(s)  // [20, 30, 40]
fmt.Println(len(s), cap(s)) // 3, 4
```
- `low` — начальный индекс (включительно)
- `high` — конечный индекс (не включительно)
- `len = high - low`
- `cap = cap(исходного) - low` (для слайса) или `len(массива) - low`
#### Полная форма `[low:high:max]`
```go
s := arr[1:3:4] // элементы 20, 30; len=2, cap=4-1=3
```
`max` ограничивает ёмкость: `cap = max - low`

### 3.2.4. Функция append

`append` добавляет элементы в слайс и **всегда возвращает новый слайс**.
```go
s := []int{1, 2, 3}
s = append(s, 4)           // добавляем один элемент
s = append(s, 5, 6, 7)     // добавляем несколько
s = append(s, []int{8,9}...) // добавляем другой слайс
```
#### Как работает append под капотом:

1. **Если `len(s) < cap(s)`**: новый элемент помещается на позицию `len(s)`, массив не меняется, `len` увеличивается на 1.
2. **Если `len(s) == cap(s)`**:
    - Создаётся новый массив с увеличенной ёмкостью
    - Все элементы копируются из старого массива
    - Добавляется новый элемент
    - Старый массив (если не используется другими слайсами) будет удалён сборщиком мусора

**Стратегия роста ёмкости:**

- Для небольших слайсов (меньше 1024 элементов) — удвоение
- Для больших — рост примерно на 25% (точная формула может меняться между версиями Go, но принцип остаётся)

### 3.2.5. Функция copy

`copy` копирует элементы из исходного слайса в целевой.
```go
dst := make([]int, 5)
src := []int{1, 2, 3}
n := copy(dst, src)  // n = 3, dst[0:3] = [1,2,3], остальные 0
```
Особенности:

- Копирует до тех пор, пока не достигнут конец одного из слайсов
- Корректно работает с пересекающимися слайсами
- Возвращает количество скопированных элементов

### 3.2.6. Связь слайсов с массивом и побочные эффекты

Когда несколько слайсов ссылаются на один базовый массив, изменения через один видны через другие:
```go
arr := [5]int{10, 20, 30, 40, 50}
s1 := arr[1:4]  // [20, 30, 40]
s2 := arr[2:5]  // [30, 40, 50]

s1[1] = 999      // меняем 30 на 999
fmt.Println(s2)  // [999, 40, 50] — изменение видно!
```
**Важно:** `append` может разорвать эту связь:
```go
s1 := []int{1, 2, 3}
s2 := s1[:2]     // [1, 2], len=2, cap=3
s2 = append(s2, 99) // ёмкости хватает — s1[2] изменится!

fmt.Println(s1)  // [1, 2, 99] — изменился!
```
Но если `append` вызывает перевыделение, связь разрывается:
```go
s1 := []int{1, 2, 3}
s2 := s1[:2]
s2 = append(s2, 99, 100) // превышает cap, создаётся новый массив
fmt.Println(s1)  // [1, 2, 3] — не изменился
```
### 3.2.7. Слайсы как параметры функций

При передаче слайса в функцию **копируется заголовок** (ptr, len, cap), но базовый массив остаётся общим.
```go
// Изменение элементов видно снаружи
func modify(s []int) {
    s[0] = 100
}

s := []int{1, 2, 3}
modify(s)
fmt.Println(s) // [100, 2, 3]
```

```go
// Append внутри функции — изменения могут не отразиться
func tryAppend(s []int, v int) {
    s = append(s, v)  // присваиваем локальной переменной
    fmt.Println("внутри:", s)
}

s := []int{1, 2, 3}
tryAppend(s, 4)
fmt.Println("снаружи:", s) // [1, 2, 3] — не изменился!
```
Чтобы изменения `append` отражались снаружи, нужно:

1. Возвращать новый слайс
2. Или передавать указатель на слайс

### 3.2.8. Подводные камни и советы

1. **Всегда присваивайте результат `append`** обратно в переменную слайса.
2. **Не делайте срез с выходом за пределы** — это паника.
3. **Понимайте, что слайс — это представление данных**, а не сами данные.
4. **Для оптимизации используйте `make` с предварительной ёмкостью**, если знаете ожидаемое количество элементов, чтобы избежать многократных перевыделений.

## 3.3. Мапы (map)

Мапа (map) — это встроенная хеш-таблица, реализующая ассоциативный массив (словарь) «ключ‑значение». Она позволяет быстро добавлять, искать и удалять элементы. В Go мапы используются повсеместно, поэтому важно понять их устройство и особенности.
### 3.3.1. Устройство мапы под капотом

Мапа в Go реализована как **хеш-таблица** — структура данных, которая в среднем выполняет операции за O(1). Основные компоненты:

- **hmap** — главная структура, представляющая мапу. Содержит:
    - `count` — количество элементов (len).
    - `B` — логарифм количества бакетов (число бакетов = 2^B).
    - `hash0` — случайное семя для хеш-функции (чтобы усложнить атаки).
    - `buckets` — указатель на массив бакетов (размером 2^B).
    - `oldbuckets` — указатель на старые бакеты при росте (эвакуации).
    - `nevacuate` — прогресс эвакуации.
    - и другие поля.
- **Бакет (bucket)** — это структура, содержащая:
    - `tophash[8]` — младшие байты хешей ключей (для быстрого поиска).
    - `keys[8]` и `values[8]` — массивы пар ключ-значение (до 8 пар в бакете).
    - `overflow` — указатель на следующий бакет (для разрешения коллизий).

**Как работает поиск:**

1. Вычисляется хеш ключа с помощью хеш-функции (с учётом семени `hash0`).
2. Младшие `B` бит хеша определяют номер бакета (`bucket = hash & (2^B - 1)`).
3. Старшие биты (т.н. `tophash`) используются для поиска внутри бакета: сравниваются с `tophash[i]`.
4. Если совпадение найдено, сравниваются ключи (на случай коллизии хешей).
5. При несовпадении переходим к overflow-бакетам.

**Коллизия** — ситуация, когда разные ключи дают одинаковый индекс бакета. В Go разрешается цепочкой overflow-бакетов (каждый бакет может ссылаться на следующий).

**Рост (rehashing):** когда средняя загрузка бакета превышает ~6.5 (т.е. в среднем >6.5 элементов в бакете), мапа увеличивает количество бакетов (B++). Рост происходит **инкрементально** (постепенно, при каждой операции эвакуируется пара бакетов), чтобы не блокировать выполнение надолго.

### 3.3.2. Создание мапы
```go
// 1. Через make — пустая мапа (готова к использованию)
m1 := make(map[string]int)

// 2. С подсказкой ёмкости (оптимизация)
m2 := make(map[string]int, 100)   // изначальное количество бакетов подберётся под ~100 элементов

// 3. Литерал
m3 := map[string]int{
    "alice": 30,
    "bob":   25,
}

// 4. Нулевая мапа (nil map)
var m4 map[string]int   // nil, len=0, читать можно (возвращает zero value), писать нельзя!
// m4["key"] = 42        // panic: assignment to entry in nil map
```
### 3.3.3. Чтение и запись
```go
m := make(map[string]int)

// Запись
m["answer"] = 42

// Чтение с проверкой существования
value, ok := m["answer"]
if ok {
    fmt.Println("значение:", value)
}

// Чтение без проверки — если ключа нет, вернёт zero value (для int это 0)
value2 := m["missing"]   // 0
// Но 0 может быть и реальным значением, поэтому всегда используйте ok, если ноль допустим.

// Количество элементов
fmt.Println(len(m))
```
### 3.3.4. Удаление
```go
delete(m, "answer")   // удаляет пару; если ключа нет — ничего не делает

// Проверка после удаления
if _, ok := m["answer"]; !ok {
    fmt.Println("ключ удалён")
}
```
### 3.3.5. Итерация по мапе
```go
m := map[string]int{"a": 1, "b": 2, "c": 3}

for key, value := range m {
    fmt.Println(key, value)
}

// Только ключи
for key := range m {
    fmt.Println(key)
}

// Важно: порядок итерации случаен (рандомизирован). Не полагайтесь на порядок!
// Причина: защита от хеш-дозирования (hash flooding) — злоумышленник не может предсказать порядок.
```
**Итерация с изменением мапы:** можно добавлять/удалять элементы во время итерации, но новый элемент может попасть в текущую итерацию, а может и нет (поведение определено, но не гарантировано).
### 3.3.6. Мапа как ссылочный тип

В Go мапа — это ссылочный тип. Когда вы передаёте мапу в функцию, копируется указатель на `hmap`, а не сами данные. Поэтому изменения внутри функции видны снаружи.
```go
func addElement(m map[string]int, key string, val int) {
    m[key] = val
}

func main() {
    m := make(map[string]int)
    addElement(m, "foo", 42)
    fmt.Println(m["foo"]) // 42
}
```
### 3.3.7. Горутинобезопасность

**Мапы НЕ безопасны для параллельного доступа.** Если одна горутина пишет, а другая читает (или две пишут) — это data race, программа может упасть с паникой `fatal error: concurrent map writes` или повредить данные.

- **Одновременное чтение из нескольких горутин** — безопасно (если нет записи).
- **Чтение + запись** — небезопасно.
- **Две записи** — небезопасно.

**Как защитить:**

1. `sync.RWMutex` — самый распространённый способ.
2. `sync.Map` — специальная мапа, оптимизированная для сценариев с частым чтением и редкой записью (или когда ключи разные).

Пример с мьютексом:
```go
type SafeMap struct {
    mu sync.RWMutex
    m  map[string]int
}

func (sm *SafeMap) Get(key string) (int, bool) {
    sm.mu.RLock()
    defer sm.mu.RUnlock()
    val, ok := sm.m[key]
    return val, ok
}

func (sm *SafeMap) Set(key string, val int) {
    sm.mu.Lock()
    defer sm.mu.Unlock()
    sm.m[key] = val
}
```
### 3.3.8. Ограничения и особенности

1. **Ключи должны быть сравнимыми** (можно использовать `==` и `!=`).  
    Подходят: `bool`, `int`, `float`, `string`, `array`, `pointer`, `channel`, `interface` (если динамический тип сравним).  
    **Не подходят:** `slice`, `map`, `function` (они не сравнимы).
2. **Порядок не гарантируется** — мапа неупорядоченная коллекция.
3. **Нельзя взять адрес элемента мапы:** `&m["key"]` вызовет ошибку компиляции. Причина — при росте мапы адрес элемента может измениться.
4. **При удалении память не освобождается сразу** (но может быть переиспользована). Чтобы освободить память, нужно создать новую мапу и скопировать элементы.
5. **Нулевая мапа (`nil`) ведёт себя по-разному:** читать можно (получаем zero value), писать нельзя (паника).

### 3.3.9. Оптимизация

- **Предварительное выделение ёмкости** через `make(map[K]V, hint)` уменьшает количество перестроений (ростов).
- **Использование `map[string]struct{}` вместо `map[string]bool` для множества** — `struct{}` не занимает памяти, что экономит ресурсы.
```go
set := make(map[string]struct{})
set["apple"] = struct{}{}
if _, ok := set["apple"]; ok {
    fmt.Println("apple в множестве")
}
```

```go
    fmt.Println("\n=== 2. Добавление ===")
    ages["Diana"] = 28
    ages["Eve"] = 22
    fmt.Println("После добавления:", ages)

    // 3. Чтение и проверка существования
    fmt.Println("\n=== 3. Чтение с проверкой ===")
    if age, ok := ages["Alice"]; ok {
        fmt.Printf("Alice: %d\n", age)
    }
    if age, ok := ages["Zoe"]; !ok {
        fmt.Println("Zoe не найдена, возвращено значение по умолчанию:", age) // 0
    }

    // 4. Удаление
    fmt.Println("\n=== 4. Удаление ===")
    delete(ages, "Bob")
    if _, ok := ages["Bob"]; !ok {
        fmt.Println("Bob удалён")
    }

    // 5. Итерация — демонстрация случайного порядка
    fmt.Println("\n=== 5. Итерация (порядок случаен) ===")
    fmt.Print("Первый проход: ")
    for k := range ages {
        fmt.Print(k, " ")
    }
    fmt.Print("\nВторой проход: ")
    for k := range ages {
        fmt.Print(k, " ")
    }
    fmt.Println("\n(Порядок может совпадать случайно, но он не гарантирован)")

    // 6. Ссылочный тип: изменения внутри функции видны снаружи
    fmt.Println("\n=== 6. Передача мапы в функцию ===")
    modifyMap(ages)
    fmt.Println("После modifyMap:", ages)

    // 7. Проблема гонки и её решение
    fmt.Println("\n=== 7. Горутинобезопасность ===")

    // НЕБЕЗОПАСНЫЙ КОД (закомментирован, чтобы не падал)
    // unsafeMap := make(map[int]int)
    // for i := 0; i < 10; i++ {
    //     go func() {
    //         for j := 0; j < 100; j++ {
    //             unsafeMap[j] = j
    //         }
    //     }()
    //     go func() {
    //         for j := 0; j < 100; j++ {
    //             _ = unsafeMap[j]
    //         }
    //     }()
    // }
    // time.Sleep(time.Second) // скорее всего panic

    // БЕЗОПАСНАЯ ВЕРСИЯ с sync.RWMutex
    type SafeMap struct {
        mu sync.RWMutex
        m  map[int]int
    }
    sm := SafeMap{m: make(map[int]int)}

    var wg sync.WaitGroup
    for i := 0; i < 10; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for j := 0; j < 100; j++ {
                sm.mu.Lock()
                sm.m[j] = j
                sm.mu.Unlock()
            }
        }()
        wg.Add(1)
        go func() {
            defer wg.Done()
            for j := 0; j < 100; j++ {
                sm.mu.RLock()
                _ = sm.m[j]
                sm.mu.RUnlock()
            }
        }()
    }
    wg.Wait()
    fmt.Println("Безопасная мапа с мьютексом протестирована")

    // 8. Множество на основе map[string]struct{}
    fmt.Println("\n=== 8. Множество (set) ===")
    set := make(map[string]struct{})
    words := []string{"apple", "banana", "apple", "cherry", "banana"}
    for _, w := range words {
        set[w] = struct{}{} // добавляем
    }
    fmt.Print("Уникальные слова: ")
    for k := range set {
        fmt.Print(k, " ")
    }
    fmt.Println()

    // 9. Мапа со слайсами в качестве значений
    fmt.Println("\n=== 9. Мапа со слайсами ===")
    tags := make(map[string][]string)
    tags["golang"] = append(tags["golang"], "backend", "concurrent")
    tags["python"] = append(tags["python"], "data science")
    fmt.Printf("tags: %+v\n", tags)

    // 10. Демонстрация: нельзя взять адрес элемента
    // fmt.Println(&ages["Alice"]) // ошибка компиляции
    fmt.Println("\nАдрес элемента мапы нельзя взять (компилятор запрещает)")

    // 11. Nil map — чтение работает, запись — паника
    var nilMap map[string]int
    fmt.Println("\n=== Nil map ===")
    fmt.Println("Чтение из nilMap:", nilMap["anything"]) // 0
    // nilMap["key"] = 1 // panic: assignment to entry in nil map
    fmt.Println("Запись в nil map вызывает панику (закомментировано)")
}
```

### Пояснения к ключевым моментам

- **Порядок итерации случаен** — это защита от атак, основанных на предсказуемости хеш-таблиц. Реализовано с помощью случайного семени (`hash0`) и добавления случайного смещения при итерации.
- **Рост мапы** происходит постепенно (incremental evacuation), поэтому даже при большом количестве элементов операции остаются быстрыми.
- **Небезопасность для параллельного доступа** — если нужна конкурентность, используйте мьютексы или `sync.Map`.
- **Nil map** полезен, когда вы хотите явно обозначить «неинициализированную» мапу, но при записи требует инициализации через `make`.

Теперь вы знаете мапы на уровне, достаточном для реальной разработки. Следующая тема — структуры и указатели.
## 3.4. Структуры

Структуры (struct) — это составные типы данных, объединяющие несколько полей (возможно, разных типов) в одну логическую единицу. В Go структуры используются повсеместно: для моделирования сущностей предметной области, конфигураций, DTO и т.д. В отличие от классов в ООП, структуры не поддерживают наследование, но предлагают **композицию** через **встраивание (embedding)**.
### 3.4.1. Определение структуры

Структура объявляется с помощью ключевого слова `type` и фигурных скобок:
```go
type Person struct {
    Name string
    Age  int
}
```
- Поля с **заглавной буквы** — экспортируемые (публичные), видны из других пакетов.
- Поля со **строчной буквы** — неэкспортируемые (приватные), доступны только внутри своего пакета.

Поля могут быть любого типа: встроенные, другие структуры, слайсы, мапы, каналы и т.д.
### 3.4.2. Создание и инициализация структур

#### Нулевая структура (zero value)

Все поля получают свои нулевые значения (строка → "", int → 0, указатель → nil и т.д.)
```go
var p Person
fmt.Println(p.Name, p.Age) // "" 0
```
#### Литерал структуры

**Плохой способ (по порядку полей):** не читаем, ломается при перестановке полей.
```go
p := Person{"Alice", 30}
```
**Рекомендуемый способ (с именами полей):** явно указываем, какое поле каким значением заполняется.
```go
p := Person{
    Name: "Alice",
    Age:  30,
}
// Частичная инициализация: Age будет 0
p2 := Person{Name: "Bob"}
```
#### Указатель на структуру
```go
p := &Person{Name: "Charlie", Age: 25}
// p — это *Person, но доступ к полям через точку работает автоматически
fmt.Println(p.Name) // Charlie
p.Age = 26
```
### 3.4.3. Теги полей (struct tags)

Теги — это строки в обратных кавычках, которые можно прикрепить к полям структуры. Они используются библиотеками для мета-информации (сериализация, валидация, ORM).
```go
type User struct {
    ID   int    `json:"id"`
    Name string `json:"name,omitempty"` // omitempty — не выводить, если пустое
    Pass string `json:"-"`              // игнорировать при JSON
}
```
**Пример с JSON:**
```go
u := User{ID: 42, Name: "Alice", Pass: "secret"}
data, _ := json.Marshal(u)
fmt.Println(string(data)) // {"id":42,"name":"Alice"}
```
**Как работают теги под капотом:**  
Пакет `reflect` позволяет получить теги во время выполнения:
```go
t := reflect.TypeOf(User{})
field, _ := t.FieldByName("Name")
fmt.Println(field.Tag.Get("json")) // "name,omitempty"
```
### 3.4.4. Сравнение структур

Структуры **сравнимы** (операторы `==` и `!=`), если **все поля сравнимы**. Сравниваются они поэлементно.
```go
type Point struct{ X, Y int }
p1 := Point{1, 2}
p2 := Point{1, 2}
fmt.Println(p1 == p2) // true
```
Если структура содержит несравнимые типы (слайс, мапа, функция), то сама структура становится **несравнимой**:
```go
type Bad struct {
    Data []int
}
// var b1, b2 Bad; fmt.Println(b1 == b2) // ошибка компиляции
```
Альтернатива — `reflect.DeepEqual`, но он медленный и не рекомендуется для горячих путей.

### 3.4.5. Анонимные структуры

Определяются на месте без имени типа. Полезны для одноразовых группировок, тестов или парсинга JSON без создания отдельного типа.
```go
point := struct {
    X, Y int
}{10, 20}
fmt.Println(point.X) // 10

// Временный объект для ответа API
apiResponse := struct {
    Status string `json:"status"`
    Data   string `json:"data"`
}{"ok", "content"}
```
Недостаток: нельзя написать функцию, которая принимает именно этот тип (нужно использовать `interface{}` или повторять определение).

### 3.4.6. Встраивание (embedding)

В Go нет наследования, но есть **композиция через встраивание**. Вы включаете одну структуру в другую без имени поля — тогда поля и методы встроенной структуры «поднимаются» (promoted) во внешнюю структуру.
```go
type Person struct {
    Name string
    Age  int
}

type Employee struct {
    Person     // встраивание (embedding)
    Salary int
}
```
**Создание:**
```go
e := Employee{
    Person: Person{Name: "Alice", Age: 30},
    Salary: 50000,
}
```
**Доступ к полям:**
```go
fmt.Println(e.Name)   // "Alice" — поднятое поле
fmt.Println(e.Person.Name) // тоже "Alice", поле Person существует
e.Age = 31
```
**Поднятие методов:** если у `Person` есть метод `Greet()`, то `Employee` автоматически его получает.
```go
func (p Person) Greet() string {
    return "Hello, I'm " + p.Name
}

e := Employee{Person: Person{Name: "Bob"}}
fmt.Println(e.Greet()) // "Hello, I'm Bob"
```
**Переопределение:** можно определить свой `Greet()` для `Employee`, тогда он перекроет поднятый.

**Встраивание vs обычное поле:**

- Обычное поле: `type Employee struct { Person Person; Salary int }` — доступ `e.Person.Name`.
- Встраивание: синтаксический сахар для доступа `e.Name`, но поле `Person` остаётся. Удобно, когда вы хотите, чтобы внешняя структура «была» внутренней.

Встраивание также работает с интерфейсами (будет в теме про интерфейсы).
### 3.4.7. Передача структур в функции

- **По значению:** копируется вся структура. Для маленьких структур (4–16 байт) это нормально, для больших — накладно.
- **По указателю:** передаётся только адрес (8 байт на 64-битной системе). Изменения полей внутри функции видны вызывающему.
```go
func (p Person) SetAgeVal(age int) {   // получатель по значению
    p.Age = age   // не меняет оригинал
}

func (p *Person) SetAgePtr(age int) {  // получатель по указателю
    p.Age = age   // меняет
}
```
**Мутаторы (методы, изменяющие состояние) обычно используют указатели.**

**Рекомендация:** если структура большая (например, больше 64–128 байт) или вы хотите изменять её в функции, передавайте указатель.

### 3.4.8. Размер и выравнивание в памяти (alignment)

Go размещает поля структуры в памяти с учётом **выравнивания** — каждое поле должно начинаться с адреса, кратного его размеру (или размеру выравнивания типа). Это делается для быстрого доступа процессора.

**Пример:**
```go
type Bad struct {
    a int8  // 1 байт
    b int64 // 8 байт
    c int8  // 1 байт
}
```
На 64-битной системе:

- `a` занимает 1 байт, после него 7 байт пустого места (padding) для выравнивания `b`.
- `b` начинается с адреса, кратного 8.
- после `b` идёт `c` (1 байт), и затем ещё 7 байт padding для выравнивания всей структуры (чтобы её размер был кратен максимальному выравниванию среди полей).

**Итоговый размер:** 24 байта.

**Оптимизация:** переставив поля по убыванию размера, можно уменьшить padding:
```go
type Good struct {
    b int64 // 8
    a int8  // 1
    
    c int8  // 1
}
// Размер: 8 + 1 + 1 + 6 байт padding = 16 байт
```
**Проверить размер можно через `unsafe.Sizeof` и `unsafe.Alignof`.**
```go
fmt.Println(unsafe.Sizeof(Bad{}))  // 24
fmt.Println(unsafe.Sizeof(Good{})) // 16
```
Это важно для экономии памяти в высоконагруженных системах (например, миллионы структур в памяти).

### 3.4.9. Конструкторы и фабрики (идиома)

В Go нет встроенных конструкторов. Вместо них принято писать функции с префиксом `New...`, которые возвращают инициализированную структуру (обычно указатель, но не обязательно).
```go
func NewPerson(name string, age int) *Person {
    return &Person{
        Name: name,
        Age:  age,
    }
}
```
### Практическое задание

**Задание:** Реализовать программу, демонстрирующую все ключевые аспекты структур.

**Решение (с подробными комментариями):**
```go
package main

import (
    "encoding/json"
    "fmt"
    "reflect"
    "unsafe"
)

// 1. Структура Address
type Address struct {
    Street     string `json:"street"`
    City       string `json:"city"`
    PostalCode string `json:"postal_code"`
}

// 2. Структура Person
type Person struct {
    Name    string  `json:"full_name"`
    Age     int     `json:"age"`
    Address Address `json:"address"`
}

// Метод для Person (будет подниматься при встраивании)
func (p Person) Greet() string {
    return fmt.Sprintf("Привет, я %s", p.Name)
}

// 3. Структура Employee с встраиванием Person
type Employee struct {
    Person               // встраивание
    JobTitle string `json:"job_title"`
}

// Employee может переопределить метод Greet (необязательно)
func (e Employee) Work() string {
    return fmt.Sprintf("%s работает как %s", e.Name, e.JobTitle)
}

// 4. Структура Manager с обычным полем Employee (не встраивание)
type Manager struct {
    Employee Employee // обычное поле
    Level    int      `json:"level"`
}

// 5. Демонстрация выравнивания
type BadOrder struct {
    a int8
    b int64
    c int8
}

type GoodOrder struct {
    b int64
    a int8
    c int8
}

func main() {
    // ========== 1. Создание экземпляров Person ==========
    fmt.Println("=== Person инициализация ===")
    p1 := Person{
        Name: "Alice",
        Age:  30,
        Address: Address{
            Street:     "Lenina 1",
            City:       "Moscow",
            PostalCode: "101000",
        },
    }
    p2 := Person{Name: "Bob"} // Age = 0, Address — нулевой
    fmt.Printf("p1: %+v\n", p1)
    fmt.Printf("p2: %+v\n", p2)

    // ========== 2. Сериализация/десериализация JSON ==========
    fmt.Println("\n=== JSON сериализация ===")
    jsonData, _ := json.MarshalIndent(p1, "", "  ")
    fmt.Println(string(jsonData))

    var p3 Person
    err := json.Unmarshal(jsonData, &p3)
    if err == nil {
        fmt.Printf("Десериализовано: %+v\n", p3)
    }

    // ========== 3. Сравнение структур ==========
    fmt.Println("\n=== Сравнение структур ===")
    p4 := Person{Name: "Alice", Age: 30, Address: p1.Address}
    if p1 == p4 {
        fmt.Println("p1 и p4 равны (все поля сравнимы)")
    } else {
        fmt.Println("p1 и p4 не равны")
    }

    // Если бы в Person был слайс, сравнение было бы невозможно.

    // ========== 4. Анонимная структура ==========
    fmt.Println("\n=== Анонимная структура ===")
    point := struct {
        X, Y int
    }{10, 20}
    fmt.Printf("Координаты: (%d, %d)\n", point.X, point.Y)

    // ========== 5. Встраивание (Employee) ==========
    fmt.Println("\n=== Встраивание (embedding) ===")
    emp := Employee{
        Person: Person{
            Name: "Charlie",
            Age:  28,
        },
        JobTitle: "Разработчик",
    }
    // Доступ к поднятым полям
    fmt.Println("Имя через поднятое поле:", emp.Name)
    fmt.Println("То же самое через Person.Name:", emp.Person.Name)
    fmt.Println("Метод Greet из Person:", emp.Greet())
    fmt.Println("Метод Work из Employee:", emp.Work())

    // ========== 6. Обычное поле (Manager) vs встраивание ==========
    fmt.Println("\n=== Обычное поле (не встраивание) ===")
    mgr := Manager{
        Employee: emp, // нужно явно обращаться к полю Employee
        Level:    3,
    }
    fmt.Println("Manager.Employee.Name:", mgr.Employee.Name)
    // mgr.Name не существует — нет поднятия!
    // mgr.Greet() тоже не поднят.

    // ========== 7. Размер и выравнивание ==========
    fmt.Println("\n=== Размер и выравнивание структур ===")
    fmt.Printf("BadOrder: размер = %d байт\n", unsafe.Sizeof(BadOrder{}))
    fmt.Printf("GoodOrder: размер = %d байт (оптимизированный порядок полей)\n", unsafe.Sizeof(GoodOrder{}))
    // Вывод обычно: BadOrder 24, GoodOrder 16

    // Можно также посмотреть смещения полей:
    type T struct {
        a int8
        b int64
    }
    var t T
    fmt.Printf("смещение a: %d, смещение b: %d\n", unsafe.Offsetof(t.a), unsafe.Offsetof(t.b))
}
```

```text
=== Обычное поле (не встраивание) ===
Manager.Employee.Name: Charlie

=== Размер и выравнивание структур ===
BadOrder: размер = 24 байт
GoodOrder: размер = 16 байт (оптимизированный порядок полей)
смещение a: 0, смещение b: 8
```
### Пояснения к важным моментам

- **Теги** позволяют управлять сериализацией без изменения имён полей в коде.
- **Сравнимость** структур важна при использовании их как ключей в мапах. Если хотя бы одно поле несравнимо, структура не может быть ключом.
- **Встраивание** — это не наследование, а композиция. Оно удобно для «примесей» (mixins), но не следует злоупотреблять, чтобы не создавать глубокие иерархии.
- **Выравнивание** критично для экономии памяти при создании миллионов объектов. Используйте сортировку полей по убыванию размера, чтобы минимизировать padding.
- **Передача по указателю** дешевле для больших структур (от ~64 байт), но создаёт дополнительное косвенное обращение. Для маленьких структур (например, с 2-3 int) передача по значению может быть быстрее.
Теперь вы готовы к следующей теме: **«Указатели»** (детальное рассмотрение) или к **«Интерфейсы»**.
