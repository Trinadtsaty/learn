## 4.1. Методы

Методы в Go — это функции, привязанные к определённому типу (чаще всего к структуре, но не только). Они позволяют организовать код в стиле "объект.действие" и являются важной частью реализации интерфейсов и инкапсуляции.

### 4.1.1. Что такое метод и его синтаксис

Метод — это функция с **особым параметром** — **получателем (receiver)**. Получатель указывается перед именем метода:

```go
func (receiver Type) MethodName(параметры) возвращаемые_типы {
    // тело
}
```

**Пример на структуре Person:**

```go
type Person struct {
    Name string
    Age  int
}

// Метод с получателем-значением (value receiver)
func (p Person) Greet() string {
    return "Hello, I'm " + p.Name
}
```

Вызов метода:
```go
alice := Person{Name: "Alice", Age: 30}
message := alice.Greet()
fmt.Println(message) // "Hello, I'm Alice"
```

### 4.1.2. Получатель значения (value receiver)

**Получатель значения** — это когда метод получает **копию** объекта. Метод может читать поля, но любые изменения затронут только копию, исходный объект останется неизменным.

```go
func (p Person) SetAgeWrong(age int) {
    p.Age = age  // меняется только копия
}

func main() {
    alice := Person{Name: "Alice", Age: 30}
    alice.SetAgeWrong(35)
    fmt.Println(alice.Age) // всё ещё 30 — не изменилось!
}
```

**Когда использовать value receiver:**
- Метод только **читает** данные, не изменяет получателя.
- Структура **маленькая** (несколько полей простых типов) — копирование дешёвое.
- Тип должен реализовать интерфейс, который ожидает методы на значениях (будет в теме интерфейсов).

### 4.1.3. Получатель указателя (pointer receiver)

**Получатель указателя** — это когда метод получает **указатель** на объект. Метод может изменять исходный объект, потому что работает с оригиналом, а не с копией.

```go
func (p *Person) SetAge(age int) {
    p.Age = age  // изменяет исходный объект
}

func main() {
    alice := Person{Name: "Alice", Age: 30}
    alice.SetAge(35)  // Go автоматически возьмёт адрес (&alice)
    fmt.Println(alice.Age) // 35 — изменилось!
}
```

**Когда использовать pointer receiver:**
- Метод **изменяет** получателя (мутатор).
- Структура **большая** — избегаем дорогого копирования.
- Для **консистентности**: если у типа уже есть методы на указателях, остальные тоже лучше сделать на указателях.
- Тип должен реализовать интерфейс, который ожидает методы на указателях.

### 4.1.4. Автоматическое преобразование между T и *T

Go предоставляет синтаксический сахар: если метод определён на указателе, но переменная имеет тип значение, Go автоматически возьмёт адрес (если переменная **адресуема**). И наоборот: если метод определён на значении, но у нас указатель, Go автоматически разыменует его.

```go
// СОЗДАЁМ ОРИГИНАЛ
alice := Person{Name: "Alice", Age: 30}

// ПЕРЕМЕННАЯ-ЗНАЧЕНИЕ
var p Person = alice     // p — это копия Алисы
p.SetAge(30)             // Go делает &p, потому что SetAge на *Person

// ПЕРЕМЕННАЯ-УКАЗАТЕЛЬ
ptr := &alice            // ptr смотрит на оригинальную Алису
ptr.Greet()              // Go разыменовывает ptr (*ptr) для вызова Greet на Person

// ГЛАВНОЕ: ptr и alice — это одно и то же
ptr.SetAge(60)
fmt.Println(alice.Age)   // 60 — оригинал изменился!
fmt.Println(ptr.Age)     // 60
```

**Важное ограничение:** автоматическое взятие адреса **не работает** для элементов мапы (map values), потому что они не адресуемы. Также не работает для некоторых других выражений.

```go
m := map[string]Person{"a": {"Alice", 30}}
// m["a"].SetAge(35) // ОШИБКА! Нельзя взять адрес элемента мапы
// Нужно так:
p := m["a"]
p.SetAge(35)
m["a"] = p
```

### 4.1.5. Методы на пользовательских типах (не только структурах)

Методы можно определять не только на структурах, но и на **любом пользовательском типе**, объявленном в текущем пакете. Это позволяет добавлять поведение к встроенным типам через «обёртку» (type definition).
#### Зачем это нужно?

Иногда у вас есть простое число, строка или слайс, и вы хотите, чтобы у них были свои методы. Например, чтобы число само могло сказать, чётное ли оно, или чтобы слайс строк умел проверять наличие элемента.

В Go **нельзя** добавить метод напрямую к встроенному типу (`int`, `string`, `[]int` и т.д.). Но можно создать **свой тип** на его основе — и у этого типа уже будут методы.
#### Пример с пользовательским типом на основе `int`

```go
// Создаём свой тип MyInt на основе встроенного int
type MyInt int

// Метод IsEven — проверяет, чётное ли число
func (mi MyInt) IsEven() bool {
    return mi%2 == 0
}

// Метод Double — возвращает удвоенное значение (тип тот же)
func (mi MyInt) Double() MyInt {
    return mi * 2
}

func main() {
    num := MyInt(42)
    fmt.Println(num.IsEven()) // true
    fmt.Println(num.Double()) // 84
}
```
#### Пример с пользовательским типом на основе слайса строк
```go
// StringSlice — это наш тип, основанный на []string
type StringSlice []string

// Contains — проверяет, есть ли target в слайсе
func (ss StringSlice) Contains(target string) bool {
    for _, s := range ss {
        if s == target {
            return true
        }
    }
    return false
}

func main() {
    fruits := StringSlice{"apple", "banana", "cherry"}
    fmt.Println(fruits.Contains("banana")) // true
    fmt.Println(fruits.Contains("grape"))  // false
}
```

#### Где это полезно на практике?

- **Добавление удобных методов к примитивам** — например, проверка валидности, форматирование, преобразование.
- **Создание типов с бизнес-логикой** — например, `type UserID int` с методом `IsValid()`.
- **Расширение стандартных коллекций** — методы `Contains`, `Filter`, `Map` для своего типа слайса.

### Важное ограничение

Нельзя определить метод на **встроенном типе напрямую** (на `int`, `string`, `[]int` и т.д.) без обёртки. Компилятор запретит:
**Ограничение:** нельзя определить метод на **встроенном типе напрямую** (int, string, []int и т.д.) без обёртки. Компилятор запретит:
```go
// ОШИБКА! Нельзя добавить метод к встроенному типу int
// func (i int) IsEven() bool { return i%2 == 0 }
```
Нужно всегда создавать свой тип (`type MyInt int`), и уже на него вешать методы.
### 4.1.6. Как это работает под капотом

Метод — это **обычная функция**, у которой первый параметр — получатель. Компилятор преобразует вызов метода в вызов функции.

```go
// Наш метод
func (p Person) Greet() string { ... }

// Компилятор фактически создаёт функцию:
func Person_Greet(p Person) string { ... }

// Вызов p.Greet() превращается в:
// Person_Greet(p)
```

**Что это значит:**
- Методы не "принадлежат" структуре в смысле виртуальной таблицы (как в C++/Java). Они просто синтаксический сахар.
- **Передача получателя по значению** — всегда копирование.
- **Передача получателя по указателю** — передаётся адрес, копирование 8 байт на 64-битной системе.

**Размещение в памяти:** методы не хранятся в каждом экземпляре структуры. Они живут в сегменте кода, как обычные функции. Таблица методов для интерфейсов создаётся отдельно (будет в теме интерфейсов).

### 4.1.7. Методы и нулевые значения (nil)

Вызов метода на `nil`-получателе допустим, если метод не обращается к полям без проверки. Это полезно для безопасной обработки.

```go
type Node struct {
    Value int
    Next  *Node
}

func (n *Node) IsEmpty() bool {
    return n == nil
}

func (n *Node) GetValue() int {
    if n == nil {
        return 0  // безопасное значение по умолчанию
    }
    return n.Value
}

func main() {
    var empty *Node // nil
    fmt.Println(empty.IsEmpty())  // true — метод вызывается на nil
    fmt.Println(empty.GetValue()) // 0 — не паникует
}
```

### 4.1.8. Экспорт методов

Как и поля, методы экспортируются, если их имя начинается с **заглавной буквы**. Сам тип может быть неэкспортируемым (строчная буква), но метод — экспортируемым. Это паттерн фабрики.

```go
package bank

type account struct { // неэкспортируемый
    balance float64
}

// Экспортируемый метод
func (a *account) Deposit(amount float64) {
    a.balance += amount
}

// Фабрика
func NewAccount() *account {
    return &account{balance: 0}
}
```

### 4.1.9. Композиция через встраивание и методы

Если структура **встраивает** другую структуру, методы встроенной структуры автоматически **поднимаются** (promoted) во внешнюю структуру.

```go
type Person struct {
    Name string
}

func (p Person) Greet() string {
    return "Hello, I'm " + p.Name
}

type Employee struct {
    Person   // встраивание
    Salary int
}

func main() {
    e := Employee{
        Person: Person{Name: "Bob"},
        Salary: 50000,
    }
    fmt.Println(e.Greet()) // "Hello, I'm Bob" — метод поднят!
}
```

**Можно переопределить метод:** если `Employee` определит свой метод `Greet()`, он перекроет поднятый.

**Отличие от наследования:** нет полиморфизма без интерфейсов. `Employee` не является подтипом `Person` для компилятора (но через интерфейсы — да).

### 4.1.10. Методы и интерфейсы (обзор)

Интерфейс удовлетворяется, если тип имеет **все методы** интерфейса. Важное различие: метод на **значении** vs на **указателе** влияет на то, какие интерфейсы реализует тип.

```go
type Greeter interface {
    Greet() string
}

// Person реализует Greeter (метод на значении)
func (p Person) Greet() string { return p.Name }

var g Greeter
g = Person{"Alice"} // OK
g = &Person{"Bob"}  // тоже OK! Go автоматически разыменовывает
```

Если метод определён **на указателе**:
```go
func (p *Person) Greet() string { return p.Name }

g = Person{"Alice"} // ОШИБКА! Person (значение) не имеет метода Greet на указателе
g = &Person{"Bob"}  // OK
```

Подробнее об этом — в следующей теме (4.2. Интерфейсы).

### Практическое задание

**Задание:** Реализовать систему банковского счёта с методами и историей транзакций.

**Решение (с подробными комментариями):**

```go
package main

import (
    "errors"
    "fmt"
    "time"
)

// 1. Банковский счёт (все поля неэкспортируемые)
type BankAccount struct {
    owner    string
    balance  float64
    currency string
}

// 2. Методы BankAccount

// Deposit — пополнение счёта (указатель, потому что изменяет баланс)
func (a *BankAccount) Deposit(amount float64) {
    if amount <= 0 {
        return // отрицательные суммы игнорируем
    }
    a.balance += amount
}

// Withdraw — снятие средств (указатель, изменяет баланс)
func (a *BankAccount) Withdraw(amount float64) error {
    if amount <= 0 {
        return errors.New("сумма должна быть положительной")
    }
    if amount > a.balance {
        return errors.New("недостаточно средств")
    }
    a.balance -= amount
    return nil
}

// Balance — получение баланса (значение, только чтение)
func (a BankAccount) Balance() float64 {
    return a.balance
}

// String — строковое представление (значение, только чтение)
func (a BankAccount) String() string {
    return fmt.Sprintf("Счёт владельца: %s, баланс: %.2f %s",
        a.owner, a.balance, a.currency)
}

// TryToChangeBalance — демонстрация, что метод на значении НЕ изменяет оригинал
func (a BankAccount) TryToChangeBalance(amount float64) {
    a.balance = amount // меняется только копия
    fmt.Println("Внутри TryToChangeBalance (значение):", a.balance)
}

// 3. Пользовательский тип TransactionType
type TransactionType int

const (
    DepositType TransactionType = iota
    WithdrawType
)

func (t TransactionType) String() string {
    switch t {
    case DepositType:
        return "Пополнение"
    case WithdrawType:
        return "Снятие"
    default:
        return "Неизвестно"
    }
}

// 4. Транзакция (вспомогательная структура)
type Transaction struct {
    Type   TransactionType
    Amount float64
    Date   time.Time
}

// 5. История транзакций (пользовательский тип на основе слайса)
type TransactionHistory []Transaction

// AddTransaction — добавляет транзакцию (указатель, изменяет слайс)
func (h *TransactionHistory) AddTransaction(tx Transaction) {
    *h = append(*h, tx)
}

// PrintAll — выводит все транзакции (значение, только чтение)
func (h TransactionHistory) PrintAll() {
    if len(h) == 0 {
        fmt.Println("История транзакций пуста")
        return
    }
    for i, tx := range h {
        fmt.Printf("%d. %s: %.2f (%s)\n",
            i+1, tx.Type, tx.Amount, tx.Date.Format("15:04:05"))
    }
}

// 6. Счёт с историей (встраивание BankAccount)
type AccountWithHistory struct {
    BankAccount // встраивание — методы BankAccount поднимаются
    History     TransactionHistory
}

// Переопределяем методы, чтобы добавлять транзакции в историю
func (a *AccountWithHistory) Deposit(amount float64) {
    a.BankAccount.Deposit(amount)
    a.History.AddTransaction(Transaction{
        Type:   DepositType,
        Amount: amount,
        Date:   time.Now(),
    })
}

func (a *AccountWithHistory) Withdraw(amount float64) error {
    err := a.BankAccount.Withdraw(amount)
    if err != nil {
        return err
    }
    a.History.AddTransaction(Transaction{
        Type:   WithdrawType,
        Amount: amount,
        Date:   time.Now(),
    })
    return nil
}

// 7. Дополнительно: пользовательский тип MyInt
type MyInt int

func (mi MyInt) IsPositive() bool {
    return mi > 0
}

func (mi MyInt) Double() MyInt {
    return mi * 2
}

func main() {
    // ========== Демонстрация BankAccount ==========
    fmt.Println("=== BankAccount ===")

    // Создаём счёт
    account := &BankAccount{
        owner:    "Alice",
        balance:  1000.0,
        currency: "USD",
    }

    // Операции
    account.Deposit(500)
    fmt.Printf("После пополнения 500: %.2f\n", account.Balance())

    err := account.Withdraw(200)
    if err != nil {
        fmt.Println("Ошибка:", err)
    }
    fmt.Printf("После снятия 200: %.2f\n", account.Balance())

    err = account.Withdraw(2000)
    if err != nil {
        fmt.Println("Ошибка (ожидаемо):", err)
    }
    fmt.Printf("Баланс остался: %.2f\n", account.Balance())

    // String метод
    fmt.Println(account.String())

    // Демонстрация автоматического преобразования
    fmt.Println("\n=== Автоматическое преобразование ===")
    var valueAcc BankAccount = *account
    valueAcc.Deposit(100) // Depost на указателе, но Go берёт адрес (&valueAcc)
    fmt.Println("valueAcc.Balance():", valueAcc.Balance()) // изменилось? Да, но это копия

    // Метод на значении НЕ меняет оригинал
    fmt.Println("\n=== Value receiver не изменяет оригинал ===")
    fmt.Printf("До TryToChangeBalance: %.2f\n", account.Balance())
    account.TryToChangeBalance(9999)
    fmt.Printf("После TryToChangeBalance (оригинал): %.2f\n", account.Balance())

    // ========== AccountWithHistory ==========
    fmt.Println("\n=== AccountWithHistory ===")

    historyAcc := &AccountWithHistory{
        BankAccount: BankAccount{
            owner:    "Bob",
            balance:  500,
            currency: "EUR",
        },
        History: TransactionHistory{},
    }

    historyAcc.Deposit(200)
    historyAcc.Withdraw(100)
    historyAcc.Withdraw(50)

    fmt.Println("Текущий баланс:", historyAcc.Balance())
    fmt.Println("История транзакций:")
    historyAcc.History.PrintAll()

    // ========== MyInt ==========
    fmt.Println("\n=== MyInt (методы на пользовательском типе) ===")

    num := MyInt(42)
    fmt.Printf("%d положительное? %t\n", num, num.IsPositive())
    fmt.Printf("%d удвоенное: %d\n", num, num.Double())

    negative := MyInt(-5)
    fmt.Printf("%d положительное? %t\n", negative, negative.IsPositive())
}
```

**Ожидаемый вывод:**

```
=== BankAccount ===
После пополнения 500: 1500.00
После снятия 200: 1300.00
Ошибка (ожидаемо): недостаточно средств
Баланс остался: 1300.00
Счёт владельца: Alice, баланс: 1300.00 USD

=== Автоматическое преобразование ===
valueAcc.Balance(): 1400.00

=== Value receiver не изменяет оригинал ===
До TryToChangeBalance: 1300.00
Внутри TryToChangeBalance (значение): 9999.00
После TryToChangeBalance (оригинал): 1300.00

=== AccountWithHistory ===
Текущий баланс: 550
История транзакций:
1. Пополнение: 200.00 (15:04:05)
2. Снятие: 100.00 (15:04:05)
3. Снятие: 50.00 (15:04:05)

=== MyInt (методы на пользовательском типе) ===
42 положительное? true
42 удвоенное: 84
-5 положительное? false
```

### Пояснения к ключевым моментам

1. **Value receiver (получатель-значение):**
   - Метод `Balance()` и `String()` — только читают данные, безопасно копировать.
   - `TryToChangeBalance()` демонстрирует, что изменения теряются.

2. **Pointer receiver (получатель-указатель):**
   - `Deposit()` и `Withdraw()` изменяют баланс — нужен указатель.
   - `AccountWithHistory.Deposit()` и `.Withdraw()` тоже на указателях, потому что они вызывают методы на указателях `BankAccount` и изменяют `History`.

3. **Автоматическое преобразование:**
   - Когда вызываем `valueAcc.Deposit(100)`, Go автоматически берёт адрес `&valueAcc`, потому что метод определён на `*BankAccount`.
   - Но `valueAcc` — это копия оригинального счёта, поэтому изменения не видны в `account`. Это важный нюанс!

4. **Методы на пользовательских типах:**
   - `TransactionType` имеет метод `String()` — удобно для вывода.
   - `TransactionHistory` — это слайс, у него есть методы `AddTransaction` (указатель, потому что `append` меняет слайс) и `PrintAll` (значение, только чтение).

5. **Встраивание (embedding):**
   - `AccountWithHistory` встраивает `BankAccount`, поэтому у него автоматически есть методы `Balance()`, `String()` и даже `TryToChangeBalance()` (хотя последний бесполезен).
   - Но мы **переопределили** `Deposit` и `Withdraw`, чтобы добавить логирование в историю. При этом внутри переопределённых методов мы вызываем `a.BankAccount.Deposit(amount)` — явно обращаемся к методам встроенной структуры.
