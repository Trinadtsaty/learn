## 4.1. Методы

Методы в Go — это функции, привязанные к определённому типу (чаще всего к структуре, но не только). Они позволяют организовать код в стиле "объект.действие" и являются важной частью реализации интерфейсов и инкапсуляции.

### 4.1.1. Что такое метод и его синтаксис

Метод — это функция с **особым параметром** — **получателем (receiver)**. Получатель указывается перед именем метода:

```go
func (receiver Type) MethodName(параметры) возвращаемые_типы {
    // тело
}
```

**Пример на структуре Person:**

```go
type Person struct {
    Name string
    Age  int
}

// Метод с получателем-значением (value receiver)
func (p Person) Greet() string {
    return "Hello, I'm " + p.Name
}
```

Вызов метода:
```go
alice := Person{Name: "Alice", Age: 30}
message := alice.Greet()
fmt.Println(message) // "Hello, I'm Alice"
```

### 4.1.2. Получатель значения (value receiver)

**Получатель значения** — это когда метод получает **копию** объекта. Метод может читать поля, но любые изменения затронут только копию, исходный объект останется неизменным.

```go
func (p Person) SetAgeWrong(age int) {
    p.Age = age  // меняется только копия
}

func main() {
    alice := Person{Name: "Alice", Age: 30}
    alice.SetAgeWrong(35)
    fmt.Println(alice.Age) // всё ещё 30 — не изменилось!
}
```

**Когда использовать value receiver:**
- Метод только **читает** данные, не изменяет получателя.
- Структура **маленькая** (несколько полей простых типов) — копирование дешёвое.
- Тип должен реализовать интерфейс, который ожидает методы на значениях (будет в теме интерфейсов).

### 4.1.3. Получатель указателя (pointer receiver)

**Получатель указателя** — это когда метод получает **указатель** на объект. Метод может изменять исходный объект, потому что работает с оригиналом, а не с копией.

```go
func (p *Person) SetAge(age int) {
    p.Age = age  // изменяет исходный объект
}

func main() {
    alice := Person{Name: "Alice", Age: 30}
    alice.SetAge(35)  // Go автоматически возьмёт адрес (&alice)
    fmt.Println(alice.Age) // 35 — изменилось!
}
```

**Когда использовать pointer receiver:**
- Метод **изменяет** получателя (мутатор).
- Структура **большая** — избегаем дорогого копирования.
- Для **консистентности**: если у типа уже есть методы на указателях, остальные тоже лучше сделать на указателях.
- Тип должен реализовать интерфейс, который ожидает методы на указателях.

### 4.1.4. Автоматическое преобразование между T и *T

Go предоставляет синтаксический сахар: если метод определён на указателе, но переменная имеет тип значение, Go автоматически возьмёт адрес (если переменная **адресуема**). И наоборот: если метод определён на значении, но у нас указатель, Go автоматически разыменует его.

```go
// СОЗДАЁМ ОРИГИНАЛ
alice := Person{Name: "Alice", Age: 30}

// ПЕРЕМЕННАЯ-ЗНАЧЕНИЕ
var p Person = alice     // p — это копия Алисы
p.SetAge(30)             // Go делает &p, потому что SetAge на *Person

// ПЕРЕМЕННАЯ-УКАЗАТЕЛЬ
ptr := &alice            // ptr смотрит на оригинальную Алису
ptr.Greet()              // Go разыменовывает ptr (*ptr) для вызова Greet на Person

// ГЛАВНОЕ: ptr и alice — это одно и то же
ptr.SetAge(60)
fmt.Println(alice.Age)   // 60 — оригинал изменился!
fmt.Println(ptr.Age)     // 60
```

**Важное ограничение:** автоматическое взятие адреса **не работает** для элементов мапы (map values), потому что они не адресуемы. Также не работает для некоторых других выражений.

```go
m := map[string]Person{"a": {"Alice", 30}}
// m["a"].SetAge(35) // ОШИБКА! Нельзя взять адрес элемента мапы
// Нужно так:
p := m["a"]
p.SetAge(35)
m["a"] = p
```

### 4.1.5. Методы на пользовательских типах (не только структурах)

Методы можно определять не только на структурах, но и на **любом пользовательском типе**, объявленном в текущем пакете. Это позволяет добавлять поведение к встроенным типам через «обёртку» (type definition).
#### Зачем это нужно?

Иногда у вас есть простое число, строка или слайс, и вы хотите, чтобы у них были свои методы. Например, чтобы число само могло сказать, чётное ли оно, или чтобы слайс строк умел проверять наличие элемента.

В Go **нельзя** добавить метод напрямую к встроенному типу (`int`, `string`, `[]int` и т.д.). Но можно создать **свой тип** на его основе — и у этого типа уже будут методы.
#### Пример с пользовательским типом на основе `int`

```go
// Создаём свой тип MyInt на основе встроенного int
type MyInt int

// Метод IsEven — проверяет, чётное ли число
func (mi MyInt) IsEven() bool {
    return mi%2 == 0
}

// Метод Double — возвращает удвоенное значение (тип тот же)
func (mi MyInt) Double() MyInt {
    return mi * 2
}

func main() {
    num := MyInt(42)
    fmt.Println(num.IsEven()) // true
    fmt.Println(num.Double()) // 84
}
```
#### Пример с пользовательским типом на основе слайса строк
```go
// StringSlice — это наш тип, основанный на []string
type StringSlice []string

// Contains — проверяет, есть ли target в слайсе
func (ss StringSlice) Contains(target string) bool {
    for _, s := range ss {
        if s == target {
            return true
        }
    }
    return false
}

func main() {
    fruits := StringSlice{"apple", "banana", "cherry"}
    fmt.Println(fruits.Contains("banana")) // true
    fmt.Println(fruits.Contains("grape"))  // false
}
```

#### Где это полезно на практике?

- **Добавление удобных методов к примитивам** — например, проверка валидности, форматирование, преобразование.
- **Создание типов с бизнес-логикой** — например, `type UserID int` с методом `IsValid()`.
- **Расширение стандартных коллекций** — методы `Contains`, `Filter`, `Map` для своего типа слайса.

### Важное ограничение

Нельзя определить метод на **встроенном типе напрямую** (на `int`, `string`, `[]int` и т.д.) без обёртки. Компилятор запретит:
**Ограничение:** нельзя определить метод на **встроенном типе напрямую** (int, string, []int и т.д.) без обёртки. Компилятор запретит:
```go
// ОШИБКА! Нельзя добавить метод к встроенному типу int
// func (i int) IsEven() bool { return i%2 == 0 }
```
Нужно всегда создавать свой тип (`type MyInt int`), и уже на него вешать методы.
### 4.1.6. Как это работает под капотом

Метод — это **обычная функция**, у которой первый параметр — получатель. Компилятор преобразует вызов метода в вызов функции.

```go
// Наш метод
func (p Person) Greet() string { ... }

// Компилятор фактически создаёт функцию:
func Person_Greet(p Person) string { ... }

// Вызов p.Greet() превращается в:
// Person_Greet(p)
```

**Что это значит:**
- Методы не "принадлежат" структуре в смысле виртуальной таблицы (как в C++/Java). Они просто синтаксический сахар.
- **Передача получателя по значению** — всегда копирование.
- **Передача получателя по указателю** — передаётся адрес, копирование 8 байт на 64-битной системе.

**Размещение в памяти:** методы не хранятся в каждом экземпляре структуры. Они живут в сегменте кода, как обычные функции. Таблица методов для интерфейсов создаётся отдельно (будет в теме интерфейсов).

### 4.1.7. Методы и нулевые значения (nil)

Вызов метода на `nil`-получателе допустим, если метод не обращается к полям без проверки. Это полезно для безопасной обработки.

```go
type Node struct {
    Value int
    Next  *Node
}

func (n *Node) IsEmpty() bool {
    return n == nil
}

func (n *Node) GetValue() int {
    if n == nil {
        return 0  // безопасное значение по умолчанию
    }
    return n.Value
}

func main() {
    var empty *Node // nil
    fmt.Println(empty.IsEmpty())  // true — метод вызывается на nil
    fmt.Println(empty.GetValue()) // 0 — не паникует
}
```

### 4.1.8. Экспорт методов

Как и поля, методы экспортируются, если их имя начинается с **заглавной буквы**. Сам тип может быть неэкспортируемым (строчная буква), но метод — экспортируемым. Это паттерн фабрики.

```go
package bank

type account struct { // неэкспортируемый
    balance float64
}

// Экспортируемый метод
func (a *account) Deposit(amount float64) {
    a.balance += amount
}

// Фабрика
func NewAccount() *account {
    return &account{balance: 0}
}
```

### 4.1.9. Композиция через встраивание и методы

Если структура **встраивает** другую структуру, методы встроенной структуры автоматически **поднимаются** (promoted) во внешнюю структуру.

```go
type Person struct {
    Name string
}

func (p Person) Greet() string {
    return "Hello, I'm " + p.Name
}

type Employee struct {
    Person   // встраивание
    Salary int
}

func main() {
    e := Employee{
        Person: Person{Name: "Bob"},
        Salary: 50000,
    }
    fmt.Println(e.Greet()) // "Hello, I'm Bob" — метод поднят!
}
```

**Можно переопределить метод:** если `Employee` определит свой метод `Greet()`, он перекроет поднятый.

**Отличие от наследования:** нет полиморфизма без интерфейсов. `Employee` не является подтипом `Person` для компилятора (но через интерфейсы — да).

### 4.1.10. Методы и интерфейсы (обзор)

Интерфейс удовлетворяется, если тип имеет **все методы** интерфейса. Важное различие: метод на **значении** vs на **указателе** влияет на то, какие интерфейсы реализует тип.

```go
type Greeter interface {
    Greet() string
}

// Person реализует Greeter (метод на значении)
func (p Person) Greet() string { return p.Name }

var g Greeter
g = Person{"Alice"} // OK
g = &Person{"Bob"}  // тоже OK! Go автоматически разыменовывает
```

Если метод определён **на указателе**:
```go
func (p *Person) Greet() string { return p.Name }

g = Person{"Alice"} // ОШИБКА! Person (значение) не имеет метода Greet на указателе
g = &Person{"Bob"}  // OK
```

Подробнее об этом — в следующей теме (4.2. Интерфейсы).

### Практическое задание

**Задание:** Реализовать систему банковского счёта с методами и историей транзакций.

**Решение (с подробными комментариями):**

```go
package main

import (
    "errors"
    "fmt"
    "time"
)

// 1. Банковский счёт (все поля неэкспортируемые)
type BankAccount struct {
    owner    string
    balance  float64
    currency string
}

// 2. Методы BankAccount

// Deposit — пополнение счёта (указатель, потому что изменяет баланс)
func (a *BankAccount) Deposit(amount float64) {
    if amount <= 0 {
        return // отрицательные суммы игнорируем
    }
    a.balance += amount
}

// Withdraw — снятие средств (указатель, изменяет баланс)
func (a *BankAccount) Withdraw(amount float64) error {
    if amount <= 0 {
        return errors.New("сумма должна быть положительной")
    }
    if amount > a.balance {
        return errors.New("недостаточно средств")
    }
    a.balance -= amount
    return nil
}

// Balance — получение баланса (значение, только чтение)
func (a BankAccount) Balance() float64 {
    return a.balance
}

// String — строковое представление (значение, только чтение)
func (a BankAccount) String() string {
    return fmt.Sprintf("Счёт владельца: %s, баланс: %.2f %s",
        a.owner, a.balance, a.currency)
}

// TryToChangeBalance — демонстрация, что метод на значении НЕ изменяет оригинал
func (a BankAccount) TryToChangeBalance(amount float64) {
    a.balance = amount // меняется только копия
    fmt.Println("Внутри TryToChangeBalance (значение):", a.balance)
}

// 3. Пользовательский тип TransactionType
type TransactionType int

const (
    DepositType TransactionType = iota
    WithdrawType
)

func (t TransactionType) String() string {
    switch t {
    case DepositType:
        return "Пополнение"
    case WithdrawType:
        return "Снятие"
    default:
        return "Неизвестно"
    }
}

// 4. Транзакция (вспомогательная структура)
type Transaction struct {
    Type   TransactionType
    Amount float64
    Date   time.Time
}

// 5. История транзакций (пользовательский тип на основе слайса)
type TransactionHistory []Transaction

// AddTransaction — добавляет транзакцию (указатель, изменяет слайс)
func (h *TransactionHistory) AddTransaction(tx Transaction) {
    *h = append(*h, tx)
}

// PrintAll — выводит все транзакции (значение, только чтение)
func (h TransactionHistory) PrintAll() {
    if len(h) == 0 {
        fmt.Println("История транзакций пуста")
        return
    }
    for i, tx := range h {
        fmt.Printf("%d. %s: %.2f (%s)\n",
            i+1, tx.Type, tx.Amount, tx.Date.Format("15:04:05"))
    }
}

// 6. Счёт с историей (встраивание BankAccount)
type AccountWithHistory struct {
    BankAccount // встраивание — методы BankAccount поднимаются
    History     TransactionHistory
}

// Переопределяем методы, чтобы добавлять транзакции в историю
func (a *AccountWithHistory) Deposit(amount float64) {
    a.BankAccount.Deposit(amount)
    a.History.AddTransaction(Transaction{
        Type:   DepositType,
        Amount: amount,
        Date:   time.Now(),
    })
}

func (a *AccountWithHistory) Withdraw(amount float64) error {
    err := a.BankAccount.Withdraw(amount)
    if err != nil {
        return err
    }
    a.History.AddTransaction(Transaction{
        Type:   WithdrawType,
        Amount: amount,
        Date:   time.Now(),
    })
    return nil
}

// 7. Дополнительно: пользовательский тип MyInt
type MyInt int

func (mi MyInt) IsPositive() bool {
    return mi > 0
}

func (mi MyInt) Double() MyInt {
    return mi * 2
}

func main() {
    // ========== Демонстрация BankAccount ==========
    fmt.Println("=== BankAccount ===")

    // Создаём счёт
    account := &BankAccount{
        owner:    "Alice",
        balance:  1000.0,
        currency: "USD",
    }

    // Операции
    account.Deposit(500)
    fmt.Printf("После пополнения 500: %.2f\n", account.Balance())

    err := account.Withdraw(200)
    if err != nil {
        fmt.Println("Ошибка:", err)
    }
    fmt.Printf("После снятия 200: %.2f\n", account.Balance())

    err = account.Withdraw(2000)
    if err != nil {
        fmt.Println("Ошибка (ожидаемо):", err)
    }
    fmt.Printf("Баланс остался: %.2f\n", account.Balance())

    // String метод
    fmt.Println(account.String())

    // Демонстрация автоматического преобразования
    fmt.Println("\n=== Автоматическое преобразование ===")
    var valueAcc BankAccount = *account
    valueAcc.Deposit(100) // Depost на указателе, но Go берёт адрес (&valueAcc)
    fmt.Println("valueAcc.Balance():", valueAcc.Balance()) // изменилось? Да, но это копия

    // Метод на значении НЕ меняет оригинал
    fmt.Println("\n=== Value receiver не изменяет оригинал ===")
    fmt.Printf("До TryToChangeBalance: %.2f\n", account.Balance())
    account.TryToChangeBalance(9999)
    fmt.Printf("После TryToChangeBalance (оригинал): %.2f\n", account.Balance())

    // ========== AccountWithHistory ==========
    fmt.Println("\n=== AccountWithHistory ===")

    historyAcc := &AccountWithHistory{
        BankAccount: BankAccount{
            owner:    "Bob",
            balance:  500,
            currency: "EUR",
        },
        History: TransactionHistory{},
    }

    historyAcc.Deposit(200)
    historyAcc.Withdraw(100)
    historyAcc.Withdraw(50)

    fmt.Println("Текущий баланс:", historyAcc.Balance())
    fmt.Println("История транзакций:")
    historyAcc.History.PrintAll()

    // ========== MyInt ==========
    fmt.Println("\n=== MyInt (методы на пользовательском типе) ===")

    num := MyInt(42)
    fmt.Printf("%d положительное? %t\n", num, num.IsPositive())
    fmt.Printf("%d удвоенное: %d\n", num, num.Double())

    negative := MyInt(-5)
    fmt.Printf("%d положительное? %t\n", negative, negative.IsPositive())
}
```

**Ожидаемый вывод:**

```
=== BankAccount ===
После пополнения 500: 1500.00
После снятия 200: 1300.00
Ошибка (ожидаемо): недостаточно средств
Баланс остался: 1300.00
Счёт владельца: Alice, баланс: 1300.00 USD

=== Автоматическое преобразование ===
valueAcc.Balance(): 1400.00

=== Value receiver не изменяет оригинал ===
До TryToChangeBalance: 1300.00
Внутри TryToChangeBalance (значение): 9999.00
После TryToChangeBalance (оригинал): 1300.00

=== AccountWithHistory ===
Текущий баланс: 550
История транзакций:
1. Пополнение: 200.00 (15:04:05)
2. Снятие: 100.00 (15:04:05)
3. Снятие: 50.00 (15:04:05)

=== MyInt (методы на пользовательском типе) ===
42 положительное? true
42 удвоенное: 84
-5 положительное? false
```

### Пояснения к ключевым моментам

1. **Value receiver (получатель-значение):**
   - Метод `Balance()` и `String()` — только читают данные, безопасно копировать.
   - `TryToChangeBalance()` демонстрирует, что изменения теряются.

2. **Pointer receiver (получатель-указатель):**
   - `Deposit()` и `Withdraw()` изменяют баланс — нужен указатель.
   - `AccountWithHistory.Deposit()` и `.Withdraw()` тоже на указателях, потому что они вызывают методы на указателях `BankAccount` и изменяют `History`.

3. **Автоматическое преобразование:**
   - Когда вызываем `valueAcc.Deposit(100)`, Go автоматически берёт адрес `&valueAcc`, потому что метод определён на `*BankAccount`.
   - Но `valueAcc` — это копия оригинального счёта, поэтому изменения не видны в `account`. Это важный нюанс!

4. **Методы на пользовательских типах:**
   - `TransactionType` имеет метод `String()` — удобно для вывода.
   - `TransactionHistory` — это слайс, у него есть методы `AddTransaction` (указатель, потому что `append` меняет слайс) и `PrintAll` (значение, только чтение).

5. **Встраивание (embedding):**
   - `AccountWithHistory` встраивает `BankAccount`, поэтому у него автоматически есть методы `Balance()`, `String()` и даже `TryToChangeBalance()` (хотя последний бесполезен).
   - Но мы **переопределили** `Deposit` и `Withdraw`, чтобы добавить логирование в историю. При этом внутри переопределённых методов мы вызываем `a.BankAccount.Deposit(amount)` — явно обращаемся к методам встроенной структуры.

Теперь вы понимаете методы в Go на глубоком уровне. Следующая тема — **«Интерфейсы»** (4.2), где мы увидим, как методы позволяют реализовывать полиморфизм.


## 4.2. Интерфейсы

Интерфейсы в Go — это мощный инструмент для абстракции и полиморфизма. Они определяют **поведение**, а не данные. В отличие от многих языков, в Go интерфейсы реализуются **неявно**, что позволяет легко создавать гибкие и слабосвязанные системы.
### 4.2.1. Неявная реализация интерфейсов (duck typing в Go)

В Go не нужно явно объявлять, что тип реализует интерфейс (нет ключевого слова `implements`). Если тип имеет **все методы**, которые требует интерфейс, он автоматически этот интерфейс реализует. Это называется **утиной типизацией**: "Если это ходит как утка и крякает как утка — это утка".

```go
// Определяем интерфейс Speaker
type Speaker interface {
    Speak() string
}

// Структура Dog
type Dog struct {
    Name string
}

// Dog реализует Speak — автоматически становится Speaker
func (d Dog) Speak() string {
    return "Гав! Я " + d.Name
}

// Структура Cat
type Cat struct {
    Name string
}

// Cat тоже реализует Speak
func (c Cat) Speak() string {
    return "Мяу! Я " + c.Name
}

func main() {
    var s Speaker
    
    s = Dog{Name: "Бобик"}   // Dog подходит
    fmt.Println(s.Speak())
    
    s = Cat{Name: "Мурка"}   // Cat подходит
    fmt.Println(s.Speak())
}
```

**Плюсы неявной реализации:**
- Слабая связанность — тип не знает об интерфейсах, которые он реализует.
- Легко добавлять новые реализации — достаточно просто реализовать методы.
- Интерфейсы определяются там, где они нужны (потребителем), а не где реализуются (поставщиком).

### 4.2.2. Определение интерфейса

Интерфейс объявляется с помощью ключевого слова `type`, затем имени, затем `interface` и списка методов в фигурных скобках.

```go
type Reader interface {
    Read(p []byte) (n int, err error)
}

type Writer interface {
    Write(p []byte) (n int, err error)
}

type Closer interface {
    Close() error
}
```

**Нулевое значение интерфейса** — `nil`. Это означает, что и таблица методов (`itab`), и указатель на данные равны `nil`.

### 4.2.3. Принцип "большие интерфейсы — плохо"

В Go предпочитают **маленькие интерфейсы** — часто всего с одним методом. Чем больше интерфейс, тем сложнее его реализовать и тем меньше он подходит для повторного использования.

**Знаменитые примеры из стандартной библиотеки:**
- `io.Reader` — один метод `Read`
- `io.Writer` — один метод `Write`
- `fmt.Stringer` — один метод `String`
- `error` — один метод `Error`

**Рекомендация:** создавайте интерфейсы там, где они нужны потребителю, а не там, где их реализуют. Это называется **интерфейсы на стороне потребителя**. Например, функция, которой нужен объект с методом `Save()`, может определить свой маленький интерфейс `type Saver interface { Save() error }` прямо в своём пакете, а не требовать, чтобы все типы реализовывали большой интерфейс.

**Цитата Роба Пайка:** "Чем меньше интерфейс, тем он мощнее" (The bigger the interface, the weaker the abstraction).

### 4.2.4. Встраивание интерфейсов (embedding)

Один интерфейс может включать (встраивать) другие интерфейсы. Это способ **композиции**, а не наследования. Интерфейс-потомок будет требовать все методы встроенных интерфейсов.

```go
type Reader interface {
    Read(p []byte) (n int, err error)
}

type Writer interface {
    Write(p []byte) (n int, err error)
}

// ReadWriter встраивает Reader и Writer
type ReadWriter interface {
    Reader
    Writer
}

// Любой тип, реализующий Read и Write, реализует ReadWriter
```

В стандартной библиотеке много таких примеров: `io.ReadWriter`, `io.ReadCloser`, `io.WriteCloser` и т.д.

### 4.2.5. Пустой интерфейс `interface{}` (any)

Пустой интерфейс — это интерфейс без методов. Поскольку у любого типа есть ноль методов, **любой тип** удовлетворяет пустому интерфейсу.

```go
var anything interface{}  // может хранить что угодно
anything = 42
anything = "hello"
anything = Dog{Name: "Rex"}

// В Go 1.18+ есть псевдоним any
var val any = "text"
```

**Применения:**
- `fmt.Println` принимает `...any`
- `json.Unmarshal` принимает `any`
- Структуры с полем `any` для хранения произвольных данных

**Проблема:** потеря типобезопасности. Чтобы достать значение обратно, нужно использовать **type assertion** или **type switch**.

### 4.2.6. Внутреннее устройство интерфейсов (iface и eface)

Это одна из самых важных частей для понимания Go. Интерфейс под капотом — это структура из двух указателей.

#### Для интерфейсов с методами (например, `Speaker`)

```go
type iface struct {
    tab  *itab      // указатель на таблицу методов
    data unsafe.Pointer // указатель на данные
}

type itab struct {
    inter *interfacetype   // информация об интерфейсе (тип, имя, методы)
    _type *_type           // информация о конкретном типе
    hash  uint32           // хеш для быстрого сравнения
    fun   [1]uintptr       // массив указателей на методы (не фиксированный размер)
}
```

**Что происходит при присваивании `var s Speaker = Dog{...}`:**

1. Компилятор проверяет, что `Dog` реализует все методы `Speaker` (здесь — `Speak`).
2. Создаётся или достаётся из глобального кэша `itab` для пары (интерфейс `Speaker`, конкретный тип `Dog`). В `itab` сохраняются указатели на методы.
3. Значение `Dog` размещается в памяти (если не убегает — может быть на стеке, иначе в куче).
4. Создаётся `iface`: `tab` указывает на `itab`, `data` указывает на данные `Dog`.

**Вызов метода через интерфейс:** `s.Speak()` → через `s.tab` находим `itab`, из него берём указатель на метод `Speak` (из массива `fun`), и вызываем его, передавая `s.data` как получатель.

#### Для пустого интерфейса `interface{}` (eface)

Пустой интерфейс проще — ему не нужна таблица методов, только информация о типе.

```go
type eface struct {
    _type *_type       // указатель на информацию о типе
    data  unsafe.Pointer // указатель на данные
}
```

### 4.2.7. Классический подводный камень: интерфейс != nil

Интерфейс равен `nil` **только когда и тип, и значение равны nil**.

```go
var err error          // nil интерфейс: (_type=nil, data=nil)
fmt.Println(err == nil) // true

var p *MyError = nil   // p — это nil-указатель конкретного типа
err = p                // теперь err: (_type=*MyError, data=nil)
fmt.Println(err == nil) // false! Интерфейс не nil!
```

**Почему так?** В `err` хранится тип `*MyError`, а не `nil`. Интерфейс знает, что он когда-то содержал указатель на `MyError` (пусть и `nil`).

**Как исправить?** Функции, возвращающие ошибку, должны возвращать явный `nil`, а не `nil` конкретного типа.

```go
func returnsNilError() error {
    // return (*MyError)(nil) // НЕПРАВИЛЬНО — интерфейс будет не nil
    return nil // ПРАВИЛЬНО
}
```

### 4.2.8. Type assertion (утверждение типа)

Type assertion позволяет извлечь конкретное значение из интерфейса.

```go
var i any = 42

// Без проверки (если тип не совпадает — panic)
num := i.(int)        // 42

// С проверкой (безопасный вариант)
value, ok := i.(string)
if ok {
    fmt.Println("Это строка:", value)
} else {
    fmt.Println("Это не строка")
}
```

### 4.2.9. Type switch (переключатель типов)

Type switch позволяет проверить несколько типов последовательно.

```go
func printType(i any) {
    switch v := i.(type) {
    case int:
        fmt.Printf("int: %d\n", v)
    case string:
        fmt.Printf("string: %q\n", v)
    case bool:
        fmt.Printf("bool: %t\n", v)
    case Dog:
        fmt.Printf("собака: %s\n", v.Name)
    default:
        fmt.Printf("неизвестный тип: %T\n", v)
    }
}
```

### 4.2.10. Интерфейсы и методы на указателях

Важное ограничение: если интерфейс требует метод, определённый **на указателе**, то переменная интерфейса может принять **только указатель**.

```go
type Greeter interface {
    Greet() string
}

type Person struct {
    Name string
}

// Метод на УКАЗАТЕЛЕ
func (p *Person) Greet() string {
    return "Hello, " + p.Name
}

var g Greeter
g = Person{"Alice"} // ОШИБКА! Person не имеет метода Greet на значении
g = &Person{"Bob"}  // OK, указатель имеет метод Greet
```

### Практическое задание

**Задание:** Реализовать программу, демонстрирующую все аспекты интерфейсов.

**Решение (с подробными комментариями):**

```go
package main

import (
    "fmt"
)

// ========== 1. Определение интерфейсов ==========

// Маленький интерфейс — один метод
type Animal interface {
    Speak() string
    Move() string
}

// Интерфейс с встраиванием (Pet включает Animal + добавляет метод)
type Pet interface {
    Animal                 // встраиваем Animal (требует Speak и Move)
    GetName() string       // дополнительный метод
}

// Интерфейсы для демонстрации встраивания
type Walker interface {
    Walk() string
}

type Runner interface {
    Run() string
}

// Athlete встраивает Walker и Runner
type Athlete interface {
    Walker
    Runner
}

// ========== 2. Реализация интерфейсов типами ==========

// Структура Dog
type Dog struct {
    Name string
}

// Dog реализует Animal.Speak
func (d Dog) Speak() string {
    return "Гав!"
}

// Dog реализует Animal.Move
func (d Dog) Move() string {
    return "Бежит на четырёх лапах"
}

// Dog реализует Pet.GetName (теперь Dog реализует и Animal, и Pet)
func (d Dog) GetName() string {
    return d.Name
}

// Dog реализует fmt.Stringer (метод String)
func (d Dog) String() string {
    return fmt.Sprintf("Dog{name=%s}", d.Name)
}

// Структура Cat
type Cat struct {
    Name string
}

func (c Cat) Speak() string {
    return "Мяу!"
}

func (c Cat) Move() string {
    return "Крадётся"
}

func (c Cat) GetName() string {
    return c.Name
}

func (c Cat) String() string {
    return fmt.Sprintf("Cat{name=%s}", c.Name)
}

// ========== 3. Реализация Athlete на структуре Person ==========
type Person struct {
    Name string
}

func (p Person) Walk() string {
    return p.Name + " идёт пешком"
}

func (p Person) Run() string {
    return p.Name + " бежит"
}

// ========== 4. Демонстрация подводного камня с nil интерфейсом ==========
type MyError struct {
    Msg string
}

func (e *MyError) Error() string {
    if e == nil {
        return "nil error"
    }
    return e.Msg
}

// НЕПРАВИЛЬНАЯ функция (возвращает nil конкретного типа)
func returnsNilMyError() error {
    var err *MyError = nil
    return err // возвращает интерфейс с типом *MyError и data=nil
}

// ПРАВИЛЬНАЯ функция
func returnsNilCorrect() error {
    return nil // явный nil интерфейса
}

// ========== 5. Функция, использующая интерфейс ==========
func MakeSound(a Animal) {
    fmt.Printf("Звук: %s, движение: %s\n", a.Speak(), a.Move())
}

// Функция для демонстрации type switch
func printTypeAndValue(i any) {
    switch v := i.(type) {
    case int:
        fmt.Printf("  int: %d\n", v)
    case string:
        fmt.Printf("  string: %q\n", v)
    case bool:
        fmt.Printf("  bool: %t\n", v)
    case Dog:
        fmt.Printf("  Dog: %+v, имя: %s\n", v, v.GetName())
    case Cat:
        fmt.Printf("  Cat: %+v, имя: %s\n", v, v.GetName())
    default:
        fmt.Printf("  неизвестный тип: %T, значение: %v\n", v, v)
    }
}

func main() {
    // ========== 1. Полиморфизм через интерфейсы ==========
    fmt.Println("=== 1. Полиморфизм ===")
    
    dog := Dog{Name: "Шарик"}
    cat := Cat{Name: "Мурка"}
    
    // Оба типа можно передать в одну функцию
    MakeSound(dog)
    MakeSound(cat)
    
    // ========== 2. Слайс интерфейсов ==========
    fmt.Println("\n=== 2. Слайс интерфейсов ===")
    
    animals := []Animal{dog, cat}
    for _, a := range animals {
        fmt.Printf("%s — %s\n", a.Speak(), a.Move())
    }
    
    // ========== 3. Pet интерфейс (встраивание) ==========
    fmt.Println("\n=== 3. Pet интерфейс (встраивание Animal) ===")
    
    var pet Pet = dog  // Dog реализует Pet (имеет Speak, Move, GetName)
    fmt.Printf("Pet: имя %s, говорит %s, движется %s\n", 
        pet.GetName(), pet.Speak(), pet.Move())
    
    // ========== 4. Пустой интерфейс (any) ==========
    fmt.Println("\n=== 4. Пустой интерфейс (any) ===")
    
    var anything any
    
    anything = 42
    fmt.Printf("anything = %v, тип = %T\n", anything, anything)
    
    anything = "hello"
    fmt.Printf("anything = %v, тип = %T\n", anything, anything)
    
    anything = dog
    fmt.Printf("anything = %v, тип = %T\n", anything, anything)
    
    // ========== 5. Type assertion ==========
    fmt.Println("\n=== 5. Type assertion ===")
    
    var a Animal = dog
    
    // Успешное извлечение Dog
    if d, ok := a.(Dog); ok {
        fmt.Printf("Успешно извлекли Dog: имя %s\n", d.Name)
    }
    
    // Неуспешное извлечение Cat (ok = false)
    if c, ok := a.(Cat); ok {
        fmt.Printf("Это Cat: %s\n", c.Name)
    } else {
        fmt.Println("a — это не Cat")
    }
    
    // ОПАСНО: без проверки (если тип не совпадёт — panic)
    // wrong := a.(Cat) // panic!
    
    // ========== 6. Type switch ==========
    fmt.Println("\n=== 6. Type switch ===")
    
    printTypeAndValue(42)
    printTypeAndValue("hello")
    printTypeAndValue(true)
    printTypeAndValue(dog)
    printTypeAndValue(cat)
    
    // ========== 7. fmt.Stringer реализация ==========
    fmt.Println("\n=== 7. fmt.Stringer ===")
    
    // Когда тип реализует String() string, fmt.Println использует его
    fmt.Println(dog)  // Выведет "Dog{name=Шарик}"
    fmt.Println(cat)  // Выведет "Cat{name=Мурка}"
    
    // Без реализации String выводилось бы {Name:Шарик}
    
    // ========== 8. Встраивание интерфейсов (Athlete) ==========
    fmt.Println("\n=== 8. Встраивание интерфейсов (Athlete) ===")
    
    alice := Person{Name: "Алиса"}
    
    var athlete Athlete = alice
    fmt.Println(athlete.Walk())
    fmt.Println(athlete.Run())
    
    // ========== 9. ПОДВОДНЫЙ КАМЕНЬ: интерфейс != nil ==========
    fmt.Println("\n=== 9. Подводный камень: интерфейс и nil ===")
    
    // Правильный nil интерфейс
    var err1 error = nil
    fmt.Printf("err1 == nil: %t\n", err1 == nil) // true
    
    // Неправильный: возвращаем nil конкретного типа
    err2 := returnsNilMyError()
    fmt.Printf("returnsNilMyError() == nil: %t\n", err2 == nil) // false!
    fmt.Printf("Значение err2: %#v, тип: %T\n", err2, err2)
    
    // Правильный способ
    err3 := returnsNilCorrect()
    fmt.Printf("returnsNilCorrect() == nil: %t\n", err3 == nil) // true
    
    fmt.Println("\nПочему err2 не nil? Потому что в интерфейсе хранится тип (*MyError) и значение (nil).")
    fmt.Println("Интерфейс равен nil, только если И тип, И значение равны nil.")
}
```

### Пояснения к ключевым моментам

1. **Неявная реализация:** структуры `Dog` и `Cat` не знают об интерфейсах `Animal` или `Pet`. Они просто реализуют методы, и компилятор сам проверяет соответствие.
2. **Маленькие интерфейсы:** `Animal` имеет всего 2 метода, `Walker` и `Runner` — по одному. Их легко реализовать и переиспользовать.
3. **Встраивание интерфейсов:** `Pet` встраивает `Animal` и добавляет `GetName()`. `Athlete` встраивает `Walker` и `Runner`. Это композиция, не наследование.
4. **Внутреннее устройство:** при присваивании `dog` интерфейсу `Animal` Go создаёт `itab` для пары (`Animal`, `Dog`) и хранит его в поле `tab` структуры `iface`. Вызов `a.Speak()` находит метод через `itab.fun`.
5. **Подводный камень с nil:** `returnsNilMyError` возвращает `error` интерфейс, который содержит тип `*MyError` и значение `nil`, поэтому `err != nil`.
6. **Type assertion и type switch** позволяют безопасно извлекать конкретные типы из интерфейсов.
